# WeaveInk：项目总览

> **"墨迹为节点，灵感为连线——编织属于你的故事宇宙。"**

**《织墨》(WeaveInk)** 是一个基于 Obsidian 双向链接哲学 + RAG 检索增强生成技术的**长篇小说创作引擎**。它不是另一个"AI 代笔工具"——它是一个**认知外骨骼**，帮你记住你创造的一切，并在你即将踏入逻辑陷阱时拉你一把。

---

## 1. 核心愿景

### 痛点：长篇小说创作中的熵增诅咒

| 痛点 | 传统创作 | 织墨方案 |
|------|---------|---------|
| **设定遗忘** | 写了 50 章后忘记第 3 章提到"主角怕水" | [[设定守护者]] 实时约束，永久记忆 |
| **逻辑断层** | A 在第 10 章死了，第 35 章又出现了 | [[图谱剧情巡检]] 自动检测连通性断裂 |
| **大纲失控** | 情节线纠缠，改一处动全身 | [[原子化大纲管理]] 章节与设定分离 |
| **上下文溢出** | 大模型对话窗口装不下 200 章的设定 | [[RAG 上下文注入]] 按需检索，只喂相关信息 |

### 永恒协议

> **"创作者负责灵感与决策，织墨负责记忆与验证。"**

这条协议定义了人机分工的边界线：AI 不替你写故事，AI 替你**记住所有设定**并在你需要时**精准呈现**。

## 2. 杀手锏功能体系

### 2.1 设定守护者 (Lore Keeper)

```
                    ┌──────────────────┐
                    │  设定守护者        │
                    │  (Lore Keeper)    │
                    └────────┬─────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
    ┌──────────┐      ┌──────────┐      ┌──────────┐
    │ 人物状态机 │      │ 世界观约束 │      │ 时间线引擎 │
    │ Character │      │  World    │      │ Timeline  │
    │  State    │      │  Rules    │      │  Engine   │
    └──────────┘      └──────────┘      └──────────┘
```

**核心防御机制**——当创作者键入新内容时：

1. **实时注入**：从本地 Markdown 知识库中检索相关人物、地点、事件节点
2. **约束校验**：将检索到的设定作为 System Prompt 注入生成管线
3. **冲突告警**：若新内容与既有设定矛盾（如"主角用左手拔剑"但设定是"主角右手持剑"），生成冲突报告

**实现关键词**：向量检索 · 命名实体识别 · 上下文窗口预算分配 · 约束松弛策略

### 2.2 图谱剧情巡检 (Graph Plot Inspection)

```
        人物节点              事件节点              地点节点
     ┌────┼────┐          ┌────┼────┐          ┌────┼────┐
     A    B    C         E1   E2   E3        P1   P2   P3
     │    │    │          │    │    │          │    │    │
     └────┼────┘          └────┼────┘          └────┼────┘
          │                    │                    │
          └────────────────────┼────────────────────┘
                               │
                    图谱全局连通性检测
```

**基于 Obsidian 双向链接的自动逻辑检测：**

- **连通性检测**：扫描作品中所有 `[[人物]]` 和 `[[事件]]` 的出度/入度。若某个关键角色突然在 10 章内"消失"（出度为 0），触发"雪藏警告"。
- **因果链追溯**：从结局节点反向 BFS，检测是否存在未连接到主线的事件孤岛。
- **时间线冲突**：通过 YAML 元数据中的时间戳字段，检测事件是否出现了时间倒流。

**实现关键词**：图数据库遍历 · n-hop 邻域检测 · YAML 时间戳解析

### 2.3 原子化大纲管理 (Atomic Outline)

```
传统大纲（耦合）                   原子化大纲（解耦）
┌─────────────────┐              ┌─────────────────┐
│ 第3章：英雄觉醒  │              │ Chapter 3 节点   │
│ ┌─────────────┐ │              │ （仅含章节指针）  │
│ │ 英雄 → 觉醒  │ │              └────────┬────────┘
│ │ 导师 → 死亡  │ │                       │ 引用
│ │ 反派 → 登场  │ │        ┌──────────────┼──────────────┐
│ └─────────────┘ │        ▼              ▼              ▼
└─────────────────┘   ┌──────────┐ ┌──────────┐ ┌──────────┐
                       │ 角色·英雄 │ │ 角色·导师 │ │ 角色·反派 │
                       │ 状态：觉醒 │ │ 状态：死亡 │ │ 状态：登场 │
                       └──────────┘ └──────────┘ └──────────┘
```

- **原子性**：每个 [[人物]]、[[地点]]、[[事件]] 作为独立的 `.md` 文件存在
- **关联性**：章节文件通过 `[[wikilink]]` 引用相关节点，而非内嵌设定
- **可重组**：修改人物设定文件 → 所有引用该人物的章节自动"感知"变更

**实现关键词**：原子节点 · 引用计数 · 变更传播 · BFS/DFS 依赖分析

## 3. 技术基石 (MVP 阶段)

### 3.1 架构总览

```
┌─────────────────────────────────────────────────────────┐
│                     Frontend (Next.js)                   │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐  │

│  │ (Obsidian   │  │ 器 (CodeMirror│  │ (冲突热力图)   │  │
│  │  Graph 风格) │  │  + 实时预览)  │  │               │  │
│  └──────┬──────┘  └──────┬───────┘  └───────┬───────┘  │
│         │                │                   │          │

┌──────────────────────────┼──────────────────────────────┐
│                     Backend (FastAPI)                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │ 知识图谱  │  │ 文档解析  │  │ RAG 调度 │  │ 冲突   │  │
│  │ 引擎     │  │ 流水线    │  │ 路由     │  │ 检测器 │  │

                          ▼                     ▼
                  ┌──────────────┐      ┌──────────────┐
                  │  Gemini API  │      │  向量数据库    │
                  │  (长文本理解) │      │  (嵌入检索)   │
                  └──────────────┘      └──────────────┘
```

### 3.2 技术选型理由

| 组件 | 选型 | 理由 |
|------|------|------|
| **Frontend** | Next.js 14+ (App Router) | SSR 渲染图谱大节点列表；CodeMirror 6 深度定制 Markdown 编辑体验 |
| **Backend** | FastAPI + WebSocket | 异步高性能；分析进度实时推送；Pydantic 模型与 JSON Schema 天然兼容 |
| **认知引擎** | Gemini 2.5 Pro | 200 万 Token 上下文窗口；原生支持长文本摘要与结构化信息抽取 |
| **向量数据库** | LanceDB (嵌入式) 或 Chroma | MVP 阶段零运维成本的嵌入式方案 |
| **图谱存储** | Obsidian 原生 `.md` + NetworkX | 源文件保持人类可读；导入 NetworkX 用于 BFS/DFS 分析 |

### 3.3 为什么不是"直接调用 ChatGPT 写小说"？

| 维度 | 传统 AI 写作 | WeaveInk |
|------|-------------|----------|
| 记忆机制 | 无状态对话 | 持久化知识图谱 + RAG |
| 一致性 | 随对话长度急剧下降 | [[设定守护者]] 约束 + 冲突检测 |
| 可控性 | 黑盒生成 | 原子化节点 → 可追溯、可回滚 |
| 长文本 | Token 窗口溢出 | 按需检索，上下文预算分配 |

## 4. 数据流转架构

### 4.1 写入管线（创作侧）

```
创作者键入 → Markdown 解析器 → YAML 抽取 + Wiki-link 解析
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              人物节点索引    事件节点索引    地点节点索引
              (JSON 化)      (JSON 化)      (JSON 化)
                    │              │              │
                    └──────────────┼──────────────┘
                                   ▼
                          向量嵌入生成 (embedding)
                                   │
                                   ▼
                            向量数据库写入
                             (LanceDB/Chroma)
```

### 4.2 RAG 上下文注入管线（查询/约束侧）

```
当前写作上下文 → 命名实体识别 (NER)
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
    人物名提取    地点名提取    事件名提取
          │           │           │
          └───────────┼───────────┘
                      ▼
              向量相似度检索 (Top-K)
                      │
                      ▼
              ┌──────────────────┐
              │ 检索出的 Markdown │
              │ 节点 → JSON 对象  │
              └────────┬─────────┘
                       │
                       ▼
              ┌────────────────────────┐
              │ 编织为 System Prompt   │
              │ "以下是当前已确认的设定" │
              │ "请确保生成内容与以下   │
              │  约束一致..."          │
              └────────────┬───────────┘
                           │
                           ▼
                    Gemini API 调用
                    （上下文窗口 =
                     用户新内容 +
                     检索到的设定 +
                     指令模板）
```

### 4.3 注入模板示例

```yaml
# 注入到 Gemini System Prompt 的结构化上下文
context:
  characters:
    - name: "林夜"
      state: "重伤、失去左臂"
      last_appeared: "第23章"
      constraints:
        - "无法使用双手武器"
        - "对火光有 PTSD 反应"
  locations:
    - name: "雾隐镇"
      status: "已被毁（第18章）"
  active_events:
    - "帝国军围城（第24章开篇）"
```

### 4.4 上下文预算分配策略

- **静态预留（20%）**：System Prompt 指令模板 + 行为约束
- **动态注入（60%）**：RAG 检索到的 Top-K 设定节点（按相关度排序）
- **弹性缓冲区（20%）**：用户当前输入的上下文、对话历史

## 5. 开发路线图 (MVP)

| 阶段 | 代号 | 目标 | 交付物 |
|------|------|------|--------|
| **P0** | `Loom` 织机 | Obsidian vault 结构化 + Markdown 解析 | YAML 抽取器、Wiki-link 索引器 |
| **P1** | `Spindle` 纺锤 | RAG 检索管线 MVP | 向量嵌入生成、语义检索 API |
| **P2** | `Shuttle` 梭子 | Gemini 上下文注入闭环 | System Prompt 编织器、约束冲突检测 |
| **P3** | `Tapestry` 织锦 | Next.js 可视化前端 | 图谱渲染、编辑器面板、巡检报告 |
| **P4** | `Loomkeeper` 织机守护者 | 端到端集成 + 异常检测 | 时间线冲突检测、角色雪藏告警 |
