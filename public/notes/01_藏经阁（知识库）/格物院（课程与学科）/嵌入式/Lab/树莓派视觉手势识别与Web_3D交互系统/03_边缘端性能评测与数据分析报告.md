# 📊 树莓派视觉手势识别与Web 3D交互系统：边缘端性能评测与数据分析报告

> [!abstract] 性能评测与优化摘要
> 本篇文档为本项目的**边缘端性能评测与数据分析报告**。系统在树莓派 3B+ (Broadcom BCM2837B0, armv7l) 边缘端对 OpenCV 特征量化与 MediaPipe Hands 混合推断模式进行了深度压力测试。通过对“抽帧跳步（Infer Skip）”灵敏度的量化分析，揭示了嵌入式视觉计算中“计算延迟”、“传输吞吐”与“芯片温控”的三维权衡曲线，并汇总了项目开发中的四大物理避坑法门。

## 🧭 一、 评测背景与测试环境

为评估该系统在边缘端低算力平台上的长期稳定性，我们对树莓派端主控程序进行了不同配置下的基准压力测试：
* **核心硬件**：树莓派 3B+ (CPU: 四核 1.4GHz Broadcom BCM2837B0，内存: 1GB LPDDR2)
* **传感器**：树莓派 500万像素 CSI 摄像头 (节点映射为 `/dev/video0`)
* **测试基准格式**：分辨率 $320 \times 240$，视频源帧率 $30\text{ fps}$，局域网以太网连接
* **图像传输**：原生 Web 异步服务器抛送 MJPEG 视频流，图像质量压缩比设置为 70

## ⚙️ 二、 性能测试指令与基准套件 (Benchmark Suite)

为评估抽帧因子 $S$ 对系统性能的边际影响，我们在树莓派端编写了专用的测试基准套件 `raspberry_perf_benchmark.py` 与自动化压力测试脚本 `run_perf_suite_csi_30fps_matrix.sh`。

### 1. 压力测试脚本 (`run_perf_suite_csi_30fps_matrix.sh`)
测试脚本循环遍历 $S \in \{15, 10, 5, 3, 2, 1\}$，每次以特定的抽帧因子持续加压测试 $15$ 秒（包括 $5$ 秒的摄像头预热/算法 Warm-up）：
```bash
#!/usr/bin/env bash
# 遍历不同抽帧步长进行混合模式基准测试
for skip in 15 10 5 3 2 1; do
  python3 /home/pi/raspberry_perf_benchmark.py \
    --camera 0 \
    --mode hybrid \
    --width 320 \
    --height 240 \
    --fps 30.0 \
    --infer-skip "$skip" \
    --model-complexity 0 \
    --max-num-hands 1 \
    --mjpeg \
    --jpeg-quality 70 \
    --duration 15 \
    --warmup 5 \
    --benchmark-csv /home/pi/perf_results_csi/skip_${skip}.csv
done
```

### 2. 性能监控核心实现 (`raspberry_perf_benchmark.py`)
在推断循环中，Python 脚本通过后台线程与 `psutil` 读取系统的硬件传感器数据：
* **CPU 使用率**：通过 `psutil.cpu_percent(interval=None)` 捕获当前进程的多核总占用。
* **CPU 温度**：通过读取树莓派系统温度虚拟文件 `/sys/class/thermal/thermal_zone0/temp` 实时计算：
  ```python
  def get_cpu_temp():
      with open("/sys/class/thermal/thermal_zone0/temp", "r") as f:
          return float(f.read().strip()) / 1000.0
  ```

> [!tip] 抽帧步长（Skip）双轨规律分析
> * **【学术轨】**：神经网络计算复杂度为 $\mathcal{O}(W \cdot H \cdot C)$。由于 CPU 算力限制，每进行一次 MediaPipe 推断都会引入约 $150\sim 200\text{ ms}$ 的硬阻塞。通过引入抽帧因子 $S$，推断频次降为 $f_{\text{infer}} = f_{\text{camera}} / S$。实测显示，当 $S \ge 10$ 时，平均循环耗时被成功压缩在 **$33.16\text{ ms}$（约合 30 FPS 的处理阈值）** 以内，同时为底层的 JPEG 编码与 WebSocket 异步并发释放了多核 CPU 空闲。
> * **【直觉轨】**：**“抓大放小”的视点折中**。网页端 3D 球体需要极高帧率的 MJPEG 背景流来保证画面不卡顿，但对于“换形状”这种低频手势操作，一秒检测 2 次（2Hz）就足够灵敏。所以我们每拍 10~15 张照片，才把其中 1 张送去神经网络“认手势”（Infer），其余照片只用来给眼睛看和计算“动了多少”（OpenCV），这样树莓派既不会发热，画面还极其流畅。

> [!important] 边缘端运行参数推荐 (Recommended Configs)
> * **日常联调混合模式**（神经网络 + 运动提取）：配置 `--infer-skip 10`。
> * **极限低功耗模式**（完全切断神经网络，纯 OpenCV 帧差面积控制）：配置 `--mode opencv`。

## 🧠 五、 嵌入式算力佯谬：识别频率越高，CPU 占用率反而越低？ (CPU Utilization Paradox)

在对抽帧步长 $S$ 进行压力测试时，我们观察到了一个极具代表性的嵌入式性能佯谬：**在不抽帧（Skip = 1，神经网络推断最高频）时，CPU 占用率（29.78%）反而低于高抽帧模式（Skip = 15 时的 34.62%），但芯片最高温度却反而上升了 $5.4^\circ\text{C}$。** 

针对这一非直观现象，本系统从操作系统调度与多核架构层面进行了双轨解密：

### 1. 双轨制机理解析

* **【直觉轨】**：
  * **高抽帧模式（Skip = 15，低负载）**：类似于一条**运转流畅、吞吐极高的快递流水线**。虽然没有重体力活（很少识别骨骼），但每秒要收发并处理 22 件快递（22 FPS）。抓图、图像差分、形态学膨胀、二进制 JPEG 压缩与网络 WebSocket 发送均在全速并发流转。由于多线程多环节齐头并进，**多核 CPU 的整体资源得到了充分调动，整体占用率较高**。
  * **无抽帧模式（Skip = 1，重负载）**：类似于**流水线上唯一的收费通道被一辆超重超载的泥头车（MediaPipe 神经网络推断）给彻底堵死**。这辆车过站需要硬性消耗约 $200\text{ ms}$ 的计算时间，导致后面的抓图、压缩、发送等线程全部被迫挂起排队。泥头车只能塞满一个通道（单核满载，受 Python GIL 限制最多贡献 $25\%$ 的多核占用），其他通道只能闲置干等。因此，**整条街的平均占用率降低了，但由于大车的发动机功率过大，导致该通道温度急剧升高**。
* **【学术轨】**：
  1. **Python GIL 限制与单核性能天花板**：树莓派 3B+ 搭载了 4 核 Broadcom 处理器。由于 Python 全局解释器锁（GIL）的存在，单线程神经网络推断在任一时刻仅能占满单个物理核心。这意味着，哪怕这颗核心处于 $100\%$ 的负荷状态，反映在 4 核系统上的整体 CPU 占用率理论上限也仅为 $\frac{100\%}{4} = 25\%$。
  2. **OpenCV 多线程库（TBB/OpenMP）的并发消退**：OpenCV 的图像矩阵差分、高斯模糊与 JPEG 图像编码（`cv2.imencode`）在底层是高度并行的多线程实现。在高帧率（22 FPS）下，每秒有 22 次高频并发像素处理，能有效分摊到 4 个核心上；而在低帧率（4 FPS）下，每秒仅调用 4 次，多核并发占比断崖式下跌，导致总 CPU 百分比反而缩水。
  3. **SIMD（NEON）指令集热密度与芯片保护性降频**：MediaPipe 包含大量的卷积矩阵运算，会密集调用 ARM 架构的 NEON 向量指令集（SIMD）。这会导致处理器的浮点运算单元（FPU）晶体管处于饱和发热状态，其热密度（单位面积发热量）远超一般的 I/O 阻塞或普通内存拷贝；随着温升逼近 $60^\circ\text{C}$，系统触发了 CPU 降频节流（Throttling），CPU 降频虽然控制了总功耗，但也使单次神经网络推断耗时拉长，加剧了系统的延迟卡顿。

## ⚠️ 六、 实验重构与系统设计痛点纪实

在大作业的调试与演进中，针对以下核心痛点进行了重构修复，是极具工程参考价值的避坑心法：

> [!warning] 痛点一：SSH Headless 模式启动时的 GUI 报段错误 (X Server Error)
> * **痛点本质**：在树莓派终端使用 SSH 远程启动代码时，因为没有挂载物理显示器，一旦程序内部执行到 OpenCV 的 `cv2.imshow`，底层的图形窗口会话（X11）就会找不到显示设备，导致系统引发段错误直接闪退。
> * **解决法门**：在 GUI 调用处封包 `try-except`。捕获到 X11 相关报错后，自动降级为 `--no-preview` 脱机流模式，切断物理窗口创建，仅将运算数据抛送给 WebSocket 与 WebGL，从而达成稳健 of headless 部署。

> [!warning] 痛点二：设备驱动独占导致的摄像头物理冲突锁死 (Device Busy Lock)
> * **痛点本质**：Linux 设备驱动节点 `/dev/video0` 具有排他独占属性。若在不同子线程或子进程里各自调用 OpenCV 读取帧进行特征计算与 MJPEG 视频流抛送，会瞬发驱动 Busy 锁死错误。
> * **解决法门**：重构为单线程帧流路由管线。每一次 `cap.read()` 读取的新帧，在内存中统一完成灰度差分、绘制渲染，然后再分别打包成二进制 JPEG 视频流与 JSON 数据流，彻底根除驱动竞争。

> [!warning] 痛点三：Web 端 MediaPipe 算力过载与内存泄漏
> * **痛点本质**：网页长时间运行于 Browser 本地摄像头模式时，高强度的 WebGL 粒子计算加上 MediaPipe Hands WebAssembly 实例的持续构建，会导致内存占用呈指数上涨，最终被浏览器 OOM 强行杀死进程。
> * **解决法门**：建立 `browserStartToken` 动态生命周期令牌机制。每当用户切出 Browser 模式时，立刻执行 Web 摄像头的 `track.stop()` 并注销释放 MediaPipe 上下文，达成即用即关的瞬时垃圾回收机制。

> [!warning] 痛点四：多维手势切换冲突与“捏合缩放”死锁 (Pinch Deadlock)
> * **痛点本质**：先前版本将“握拳（0指）”直接映射为 `heart`。当用户做双指捏合（Pinch）缩小 3D 球体时，其他手指会自然攥紧合拢，导致系统误判为“握拳”并强行闪变成爱心形态。为此引入了 `shapeLocked` 机制（间距 $<0.08$ 锁死形态切换），但这又导致用户在缩放过程中完全无法换手势，造成体验僵死。
> * **解决法门**：
>   1. **正交几何解耦**：彻底移除“握拳 $\rightarrow$ 爱心”的直接映射。将爱心重构为**倒挂比心 (Downward Four-Finger Heart)**，土星重构为**OK 手势**，与水平捏合缩放完全正交。
>   2. **锁特权逃逸**：在 `shapeLocked = true` 期间，唯有检测到强特征的 *Downward Heart* 或 *OK* 时，才允许破锁切换形状，其余姿态一律屏蔽。
>   3. **物理阈值回归**：将全局判定阈值还原为严格的基准值（$R_{\text{thresh}} = 1.3$, $S_i = 0.80$），并结合前置的“滑动窗口投票（Voting Window）”过滤机制，必须在连续 $N$ 帧（如 8 帧中占 6 帧以上）判定为同一手势时才执行切换，彻底消除了形态过渡阶段的视觉抖动。
> 
> > [!warning] 痛点五：CSI 与 USB 摄像头设备索引偏移与 sudo 运行环境隔离
> > * **痛点本质**：
> >   1. **设备索引偏移**：树莓派挂载多个 `/dev/video*` 设备时，排线 CSI 摄像头（Sunny CSI）和外接 USB 摄像头的路径索引不同。官方 CSI 摄像头默认绑定在 `--camera 0`，而 USB 摄像头通常在 `--camera 1`；一旦指令中索引指定错误，会报错 `can't open camera by index`。
> >   2. **sudo 权限隔离**：由于 OpenCV 通常是用当前用户（如 `pi`）本地安装在 `~/.local/lib/python3.*/site-packages` 目录下，若在运行脚本时误加了 `sudo python3`，系统会切到 `root` 用户的 Python 运行环境，从而导致 `ModuleNotFoundError: No module named 'cv2'` 报错。
> > * **解决法门**：
> >   一句话心法：**“CSI 用 camera 0，Python 运行不要加 sudo”**。在运行服务端之前，先通过 `vcgencmd get_camera` 确认硬件状态为 `supported=1 detected=1`（硬件正常识别），然后直接在当前普通用户权限下运行，避免 root 环境导致的包路径丢失：
> >   `python3 raspberry_gesture_server.py --camera 0 --mode hybrid --width 320 --height 240 --fps 30 ...`
