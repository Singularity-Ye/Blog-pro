# ACTION表：决策移进还是归约的操作手册

> English: **ACTION Table**

**ACTION 表** 说白了就是自底向上分析程序中**“专门对外联络、决定下一步做什么的台前执行总监”**。它根据当前卡车里贴着什么状态标签（栈顶状态，行），以及传送带上接下来送来了什么符号（当前输入终结符，列），来下达四种具体的指挥指令（移进、归约、接受或报错）。

---

## 🚗 直觉比喻：台前执行总监 (The Frontstage Director)

> [!NOTE]
> 在整个编译大戏中， **ACTION 表** 扮演着 **“台前执行总监”** 的角色：
>
> *   **面对的观众（列）** $\rightarrow$ **终结符 (Token)** ：它只跟台下的普通观众（即代码中的关键字、标识符、运算符以及代表演出结束的 $\text{＄}$ 符号）打交道。
> *   **当前台上的布景（行）** $\rightarrow$ **分析栈顶状态** ：当前剧场所处的历史状态。
> *   **戏剧指令（表项值）** $\rightarrow$ **四种指令** ：针对特定布景和出场观众，总监给演员们下达明确的指令（移进、归约、接受或当场终止报错）。

---

## 形式化定义与结构

ACTION 表是一个由行和列组成的二维矩阵：
*   **行坐标 (Row)** ：代表活前缀识别 DFA 的当前状态集编号（如 $0, 1, 2, \dots$）。
*   **列坐标 (Column)** ：代表文法中的所有 **终结符 (Terminals)** ，外加一个输入流结束符 **$\text{＄}$** 。

### 示例图表

设分析表局部如下：

| 状态 | id | + | * | $ |
| :---: | :---: | :---: | :---: | :---: |
| **$0$** | $s_5$ | | | |
| **$1$** | | $s_6$ | | $\text{acc}$ |
| **$2$** | | $r_2$ | $s_7$ | $r_2$ |

> **符号释义** ：
> *   $s_5$ 表示：移进终结符并跳转到状态 $5$。
> *   $r_2$ 表示：使用第 $2$ 条产生式进行归约。
> *   $\text{acc}$ 表示：接受，解析成功。
> *   **空白格子** 表示：出现语法错误，抛出异常。

## 🎭 四大核心动作详解

总监（ACTION 表）能够发出的指令共有四种：

### 1. 移进 (Shift，缩写为 $s_j$)
*   **物理行为** ：读入下一个终结符 $a$，将它和与之对应的目标状态编号 $j$（即 $s_j$ 中的 $j$） **压入分析栈顶** 。
*   **舞台效果** ：邀请台下的观众 $a$ 走上舞台，并切换背景幕布为第 $j$ 幕。
*   **关联概念** ：[[移进]]。

### 2. 归约 (Reduce，缩写为 $r_k$)
*   **物理行为** ：
    1.  确定第 $k$ 条产生式为 $A \to \gamma$。
    2.  从分析栈中弹出长度为 $2 \times |\gamma|$ 的元素（即右部符号串及其对应的状态符号）。
    3.  将新生成的左部非终结符 $A$ 压入栈顶，并拿着刚才露出的历史栈顶状态，去查 **[[GOTO表]]** 决定压入哪个新状态。
*   **舞台效果** ：台上的几位小演员表演完毕，向幕后鞠躬下台。换成代表他们组合的“大咖”非终结符 $A$ 上台展示。
*   **关联概念** ：[[归约]]。

### 3. 接受 (Accept，缩写为 $\text{acc}$)
*   **物理行为** ：整个输入序列被成功读完且无冲突，分析栈仅剩起始符号与初始状态，分析器宣布程序解析成功。
*   **舞台效果** ：大戏圆满落幕，全场鼓掌欢呼。
*   **关联概念** ：[[增广文法]]。

### 4. 报错 (Error，通常在表里留白)
*   **物理行为** ：分析栈顶状态在面对当前输入字符时，在 ACTION 表中查到的是 **空白格** 。分析器立刻终止执行，启动语法错误恢复机制或直接输出报错信息。
*   **舞台效果** ：演员忘词或在台上发生穿帮，导演立刻叫停。

## ⚠️ 表格中的冲突表现

在构建分析表或编译 Bison 代码时，如果文法本身存在二义性，ACTION 表格中就会出现 **同一个单元格被填入两个不同动作** 的异常死锁（Bison 会报错提示）：

*   **移进-归约冲突 (Shift/Reduce Conflict)** ：在格子 `ACTION[i, a]` 里同时存在 $s_j$ 和 $r_k$。面对输入 $a$，分析表无法决断是该先吃进字符（移进 $s_j$）还是先折叠右部（按第 $k$ 条产生式归约 $r_k$）。
*   **归约-归约冲突 (Reduce/Reduce Conflict)** ：在格子 `ACTION[i, a]` 里同时存在 $r_p$ 和 $r_q$。面临输入 $a$，分析栈顶的内容同时符合两条产生式右部，无法决断使用哪条产生式折叠。

---

## 🛠️ 冲突的消解：表条目的改写与重构

在 Bison/Yacc 编译阶段，分析表生成器通常使用结合性与优先级指令来静态改写（Rewrite）ACTION 表的条目，强制消解冲突以保证分析表单值确定：

### 1. 结合性与优先级指令的消解原理
以二义性产生式 $E \to E + E \mid \text{id}$ 为例，当栈内为 `E + E`，且下一个待移进字符为 `+` 时，分析器面临两难决策：
*   **动作 1（归约）**：按 $E \to E + E$ 将已有的 `E + E` 归约为 `E`。
*   **动作 2（移进）**：将新读入的 `+` 移进栈内。

若我们声明了 `+` 为左结合（`%left '+'`）：
1. 分析表生成器比较“当前产生式 $E \to E + E$ 的优先级”（取决于最右侧运算符 `+`）与“待移进字符 `+` 的优先级”。
2. 两者优先级相同，且同为**左结合**。
3. 按照左结合原则（左侧优先计算），分析表生成器会**主动丢弃**移进动作 $s_j$，在 `ACTION` 单元格中**仅填入**归约动作 $r_k$。冲突在建表期即被抹平消解。

### 2. 规则优先级重写（`%prec`）
在遇到负号（一元减法）与减号冲突等情况时，可通过 `%prec` 赋予产生式与终结符不同的优先级。例如：
`expr: '-' expr %prec UMINUS`
这会强制分析表生成器在遇到负号时，使用 `UMINUS` 的超高优先级改写 `ACTION` 表项，消解减法移进的冲突。

## 🚨 空白错误槽与错误恢复机制 (Error Recovery)

在 ACTION 表中，所有未被填入任何动作的 **空白格子 (Blank Cell)** 均为 **语法错误接口**。

### 1. 报错的触发
当分析器处于栈顶状态 $s$，读入字符 $a$，若查表 `ACTION[s, a]` 得到的是空白，分析器立刻判定输入流出现语法错误。

### 2. Bison 中的 `error` 标记物理恢复机制
如果我们在规格说明文件中写入了带有 `error` 终结符的容错产生式（如 `stmt: error ';'`），Bison 生成的 `yyparse()` 会执行如下物理恢复：
1. **弹栈清退**：沿着分析栈不断弹出状态，直到露出一个 **能够移进特殊符号 `error`** 的历史状态。
2. **符号移进**：将特殊符号 `error` 压入栈顶，并跳转到该状态。
3. **丢弃输入**：继续读入并丢弃输入流中的 Token，直到遇到能使分析器恢复工作的 **同步 Token（如分号 `;` 或右括号 `}`）**。
4. **重新同步**：一旦在 ACTION 表中发现针对同步 Token 存在合法的移进动作，分析器便重归正轨，继续工作。

## 🔄 ACTION 表与 GOTO 表的差异分工

| 维度特性 | ACTION 表 (台前总监) | GOTO 表 (幕后调度) |
| :--- | :--- | :--- |
| **列的性质** | **终结符** ($a \in T$) + 结束符 $\$$ | **非终结符** ($A \in N$) |
| **查表时机** | 每次 **读入新终结符** 时必查 | 仅在 **归约动作完成** 、弹栈露出历史状态后必查 |
| **输出项** | $s_j$ (移进), $r_k$ (归约), $\text{acc}$, 空白 | 目标状态编号 $j$ |
| **冲突可能性** | **存在** （Shift/Reduce 或 Reduce/Reduce） | **绝无可能** （DFA 转移具有确定性） |
