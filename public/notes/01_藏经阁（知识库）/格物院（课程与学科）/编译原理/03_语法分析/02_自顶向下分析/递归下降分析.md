# 递归下降分析：一层函数吃掉一层语法结构

> English: **Recursive Descent Parsing**

> [!info] 学习定位
> - **类型**：分析方法
> - **来源章节**：Chapter 4
> - **使用章节**：Chapter 4
> - **主要用途**：用递归函数实现 top-down parsing，含 left recursion 时会无限递归
> - **相关题型**：LL(1) 综合题、grammar 适用性判断

> [!NOTE] 双轨直觉：为每个非终结符配一个专属客服函数
> - **客服函数分工**：每个非终结符都是一个专属的**“客服人员”**（在代码里对应一个子函数，如 `parse_E()`, `parse_T()`）。
> - **派单与通话过程**：当非终结符客服 `E` 开始接待用户时，它根据手里的产生式剧本，发现需要先处理 `T` 业务，于是打电话给 `T` 客服：『你先帮我搞定这个』。等 `T` 客服搞定挂断电话后，`E` 客服继续处理剩下的字母。如果遇到具体的终结符字母，客服就当场和当前的输入流字符进行核对，对上了就消耗掉，对不上就立刻报错。
> - **死循环危机**：如果文法中含有左递归（如 $A \to A\alpha$），客服 `A` 一上班，干的第一件事就是打电话呼叫自己。这会产生一个**永不挂断、无限派单给自己**的“死循环通话”，瞬间把公司的客服电话机占满（系统调用栈溢出 Stack Overflow），导致整个系统崩溃。

## 定义

递归下降分析是一种自顶向下语法分析方法，为每个非终结符编写一个**递归函数**，函数体按照产生式右部的结构调用其他函数或匹配终结符。

## 核心特征

- 每个非终结符对应一个函数
- 函数内部用 if-else 选择产生式
- 可能需要回溯（若文法不是 LL(1)）
- 若文法是 LL(1)，可以做成**无回溯递归下降**（即预测分析的代码实现）

## 伪代码示例

```python
def parse_exp():
    parse_term()
    parse_exp_prime()

def parse_exp_prime():
    if lookahead == '+':
        match('+')
        parse_term()
        parse_exp_prime()
    # else: ε-production, do nothing
```

## 在题目里怎么用

递归下降分析一般不会要求完整手写代码，但会考察：

- 能否根据 grammar 写出每个 nonterminal 对应的函数
- 能否判断某个 grammar 是否适合无回溯递归下降
- 能否说明 left recursion 为什么会导致递归下降**无限递归**

如果文法含有 left recursion（$A \to A\alpha \mid \beta$），则 `parse_A()` 会一进入就再次调用 `parse_A()`，导致无限递归。

## 直接相关易错点

- 递归下降分析 ≠ LL(1) parsing table。递归下降通常是**代码驱动**，LL(1) parser trace 通常是 **table driven**
- 含 left recursion 的 grammar 不能直接写成递归下降 parser
- 预测分析可以看作无回溯递归下降的一种系统化形式
