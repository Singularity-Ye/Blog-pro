## 1. Introduction

This project implements a scientific calculator parser using Flex and YACC/Bison. The lexer from Lab 1 is reused to recognize numbers, identifiers, operators, functions, constants, and memory commands. The main task of Lab 2 is to design the grammar, construct an Abstract Syntax Tree (AST), resolve grammar conflicts, and evaluate expressions.

YACC/Bison was selected because it is closely related to the LALR parsing theory studied in this course. It provides conflict reports through the `.output` file and supports semantic actions, which are necessary for AST construction. Compared with modern alternatives such as ANTLR, Bison is more suitable for this lab because it directly demonstrates shift/reduce conflicts, precedence declarations, and Flex integration.

## 2. Grammar Design

The calculator grammar supports multiple statements separated by newlines or semicolons. Each statement can be an expression, an assignment, or a memory operation.

### 2.1 Full BNF Grammar

```
program      → ε
             | program statement terminator
             | program terminator

terminator   → NEWLINE
             | SEMICOLON

statement    → expr
             | ID '=' expr
             | MC
             | M_PLUS expr
             | M_MINUS expr

expr         → NUMBER
             | ID
             | PI_CONST
             | E_CONST
             | MR
             | expr '+' expr
             | expr '-' expr
             | expr '*' expr
             | expr '/' expr
             | expr '^' expr
             | expr POW expr
             | '-' expr                   /* %prec UMINUS */
             | '(' expr ')'
             | SIN '(' expr ')'
             | COS '(' expr ')'
             | TAN '(' expr ')'
             | LOG '(' expr ')'
             | LN  '(' expr ')'
             | LOG10 '(' expr ')'
```

### 2.2 Design Rationale

The rule `statement → ID '=' expr` handles variable assignment. The identifier is stored in the symbol table during evaluation. The token `MR` appears inside `expr` because it returns a numeric value, making it semantically equivalent to a number constant. `M_PLUS` and `M_MINUS` are statement-level rules because they modify memory state and do not produce a value that can be used in further expressions. Both `^` and `POW` are supported as exponentiation operators with identical semantics.

The grammar uses a single `expr` nonterminal for all expression types, relying on Bison's precedence and associativity declarations rather than splitting expressions into multiple levels (such as `term` and `factor`). This keeps the grammar file compact and shifts the disambiguation burden to `%left`, `%right`, and `%prec` declarations.

## 3. Conflict Analysis

### 3.1 Origin of Conflicts

The initial expression grammar caused shift/reduce conflicts because it used flat rules such as `expr → expr '+' expr` and `expr → expr '*' expr` without any precedence declarations. For the input `2 + 3 * 4`, the parser could not determine whether to reduce `2 + 3` first (giving `(2+3)*4 = 20`) or shift `*` first (giving `2+(3*4) = 14`).

### 3.2 Resolution

The following precedence declarations were added to the Bison file:

```yacc
%left '+' '-'
%left '*' '/'
%right '^' POW
%right UMINUS
```

The declaration `%left '+' '-'` makes addition and subtraction left-associative, so `10 - 3 - 2` is parsed as `(10 - 3) - 2 = 5`. The declaration `%left '*' '/'` gives multiplication and division higher precedence than addition because it appears later in the declaration list. The declaration `%right '^' POW` makes exponentiation right-associative, so `2 ^ 3 ^ 2` is parsed as `2 ^ (3 ^ 2) = 512`. The rule `'-' expr %prec UMINUS` assigns a precedence level to unary minus that is higher than all binary operators, preventing the parser from confusing `-5 - 3` with `-(5 - 3)`.

### 3.3 Verification

After adding these declarations, Bison resolved all expected expression-related conflicts. The `.output` file generated by `bison -v` was inspected to confirm that no unexpected shift/reduce or reduce/reduce conflicts remained.

## 4. AST Design

### 4.1 Node Types

The parser constructs an Abstract Syntax Tree during parsing. Each grammar rule uses semantic actions to create AST nodes. The project uses a single `Node` structure with a `NodeKind` enum to distinguish node types.

| NodeKind | Stored Data | Children | Example |
|------|------|:--:|------|
| `NODE_NUMBER` | `double value` | 0 | `3.14` |
| `NODE_IDENTIFIER` | `char *name` | 0 | `x` |
| `NODE_BINARY` | `char *op` | 2 | `a + b` |
| `NODE_UNARY` | `char *op` | 1 | `-x` |
| `NODE_FUNCTION` | `char *func_name` | 1 | `sin(x)` |
| `NODE_ASSIGN` | `char *name` | 1 | `x = 5` |
| `NODE_MEMORY` | `MemOpType mem_op` | 0 or 1 | `M+ 10`, `MR` |

### 4.2 Construction

AST construction functions include `ast_number`, `ast_identifier`, `ast_binary`, `ast_unary`, `ast_function`, `ast_assign`, and `ast_memory`. Each function allocates a `Node`, sets its type and data fields, and links its child nodes.

For example, the grammar rule `expr '+' expr` executes:

```c
$$ = ast_binary("+", $1, $3);
```

Here `$1` and `$3` are child AST nodes produced by earlier reductions, and `$$` stores the newly created parent node. This mechanism is a direct implementation of synthesized attributes: child node values propagate upward during bottom-up parsing.

### 4.3 Evaluation

The function `ast_eval` recursively evaluates the tree. It returns numeric values for number nodes, looks up identifiers in the symbol table, applies binary and unary operators, calls mathematical library functions (sin, cos, tan, log, ln, log10), performs variable assignments, and reads or updates the calculator memory register.

### 4.4 Memory Management

The function `ast_free` recursively releases left and right child nodes before freeing the current node. Dynamically allocated strings—identifier names and function names stored with `strdup`—are freed as part of this process.

## 5. Testing

Seven test cases were used to verify the parser against the functional requirements.

| No. | Input | Expected AST | Expected Result | Requirement |
|:--:|------|------|------|------|
| 1 | `(1 + 2) * 3` | Root `*`, left child `+` | 9 | Parentheses and basic arithmetic |
| 2 | `sin(cos(0.5))` | Root `sin`, child `cos` | ≈ 0.7692 | Nested trigonometric functions |
| 3 | `2 ^ 3 * 4` | Root `*`, left child `^` | 32 | Power and multiplication precedence |
| 4 | `log10(100) + ln(e)` | Root `+`, children `log10` and `ln` | 3 | Logarithmic functions and constants |
| 5 | `x = 5; y = x + 2; y` | Assignment chain ending in identifier `y` | 7 | Variable assignment and recall |
| 6 | `MC; M+ 10; M+ 5; MR * 2` | Memory nodes followed by `*` | 30 | Memory operations |
| 7 | `0b1010 + 0x10 + 0o7` | Nested `+` tree with values 10, 16, 7 | 33 | Binary, octal, hex tokens from Lab 1 |

Test 3 confirms that exponentiation binds tighter than multiplication: `2 ^ 3 * 4` produces `(2^3) * 4 = 32`, not `2 ^ (3*4) = 4096`. Test 5 confirms that variable assignment persists across semicolon-separated statements. Test 6 confirms that `M+` accumulates into memory and `MR` reads the accumulated value.

## 6. Conclusion

The parser correctly handles the required scientific calculator expressions. It parses arithmetic expressions, trigonometric and logarithmic functions, mathematical constants, variables with assignment, and memory operations. The AST is constructed during parsing and printed before evaluation.

The initial grammar design was straightforward but produced shift/reduce conflicts due to flat expression rules. Adding precedence and associativity declarations resolved all expected conflicts without restructuring the grammar into multiple nonterminal levels.

Working with Bison provided concrete experience with LALR parsing. The `.output` file made it possible to examine exactly which parser states contained conflicts and which tokens triggered them. Semantic actions turned out to be a practical application of synthesized attributes: values flow from child nodes to parent nodes during reduction, which mirrors the attribute evaluation order in S-attributed grammars.

The main difficulties encountered were: (1) shift/reduce conflicts from ambiguous expression rules, (2) distinguishing unary minus from binary subtraction using `%prec UMINUS`, (3) ensuring consistent `yylval` types between the Flex lexer and Bison parser, and (4) managing memory for dynamically allocated strings in AST nodes. Each of these was resolved through Bison precedence declarations, the `%prec` mechanism, careful `yylval` field assignments in the lexer, and a recursive `ast_free` function.
