# 依赖图与属性求值顺序：谁依赖谁、谁先算的排队图

> [[属性文法|属性文法]]只定义**每个属性怎么算**（以产生式为粒度的语义方程）。依赖图负责回答：在整棵语法树上，**按什么顺序执行所有这些语义方程**才能保证每个属性在被使用之前已经计算完毕。

---

## 一、问题定义

### 1.1 为什么需要顺序

一条语义规则的形式：

```
X.a = f(Y.b, Z.c, ...)
```

编译器要执行这条规则，必须先知道 `Y.b`、`Z.c` 的当前值。这不是单条规则的问题——一棵语法树上有**成百上千个属性实例**，它们的求值顺序不能随意。

### 1.2 形式化问题

**输入**：语法树 T，属性文法 AG = (G, A, R)，以及每个属性实例 of 语义规则集合。

**输出**：所有属性实例的一个求值顺序 `a₁, a₂, ..., aₖ`，满足：

```
∀ 语义规则 X.a = f(..., Y.b, ...)：
  Y.b 在序列中的位置 < X.a 在序列中的位置
```

等价于：把依赖图做**拓扑排序**。

---

## 二、依赖图的构造算法

### 2.1 定义

对给定语法树 T 和属性文法 AG，**依赖图 (Dependency Graph)** 是一个有向图：

```
节点：T 中每一个文法符号节点的每一个属性实例 (X.a)
边：Y.b → X.a  当且仅当存在一条语义规则 X.a = f(..., Y.b, ...)
```

箭头方向：**被依赖的属性 → 依赖它的属性**（先算的指向后算的）。

### 2.2 构造算法

```
输入：语法树 T，属性文法 AG
输出：依赖图 DG = (V, E)

1. V = { 每个语法树节点的每个属性实例 }
2. E = ∅
3. For 语法树 T 的每个内部节点 n（对应产生式 p: X₀ → X₁...Xₖ）：
     For 产生式 p 的每条语义规则 r：
       设 r 的形式为 attr₀ = f(attr₁, attr₂, ..., attrₘ)
       For j = 1 to m：
          E = E ∪ { (attrⱼ → attr₀) }   // attrⱼ 被 attr₀ 依赖
4. return DG
```

### 2.3 依赖边方向的直觉

```
依赖图的边 = "你先算，我才能算"

Y.b ──→ X.a  读作：「Y.b 算完了 → 才可以算 X.a」
```

这和属性语义方程的定义完全对应——`X.a` 出现在 f 左侧，`Y.b` 出现在 f 的右侧参数中。

## 三、例题一：数字求值的依赖图与拓扑排序

### 3.1 语义规则

```
number₁ → number₂ digit
  number₁.val = number₂.val × 10 + digit.val

number → digit
  number.val = digit.val

digit → 0 ... digit → 9
  digit.val = 常量
```

### 3.2 为 345 构造的语法树

```
          number₀ (345)
         /            \
    number₁ (34)      digit₂ (5)
    /        \
number₂ (3)  digit₁ (4)
    |
  digit₀ (3)
```

### 3.3 属性实例与语义规则实例（在具体语法树上展开）

| 节点 | 产生式 | 语义规则实例 |
|------|--------|------------|
| digit₀ | `digit → 3` | `digit₀.val = 3` |
| number₂ | `number → digit` | `number₂.val = digit₀.val` |
| digit₁ | `digit → 4` | `digit₁.val = 4` |
| number₁ | `number → number digit` | `number₁.val = number₂.val × 10 + digit₁.val` |
| digit₂ | `digit → 5` | `digit₂.val = 5` |
| number₀ | `number → number digit` | `number₀.val = number₁.val × 10 + digit₂.val` |

### 3.4 依赖图

```
digit₀.val ──→ number₂.val
digit₁.val ──→ number₁.val
number₂.val ──→ number₁.val
digit₂.val ──→ number₀.val
number₁.val ──→ number₀.val
```

### 3.5 拓扑排序得到合法计算顺序

```
digit₀.val    (1)
number₂.val   (2)
digit₁.val    (3)
number₁.val   (4)
digit₂.val    (5)
number₀.val   (6)
```

注意到：这就是自底向上 LR 分析 of 归约顺序。纯 [[S-属性文法|S-属性文法]]的依赖图天然是**无环的**，且拓扑顺序恰好对应自底向上的归约历程。

## 四、例题二：声明类型传播的依赖图（混合属性方向）

### 4.1 语义规则

```
decl → type var-list
  var-list.dtype = type.dtype

type → int
  type.dtype = integer    (常量)

var-list₁ → id , var-list₂
  id.dtype = var-list₁.dtype
  var-list₂.dtype = var-list₁.dtype

var-list → id
  id.dtype = var-list.dtype
```

### 4.2 为 `float x, y` 构造的语法树

```
              decl
         /           \
     type            var-list₀
      |           /           \
    float     id(x)         var-list₁
                              |
                            id(y)
```

### 4.3 语义规则实例

| 节点 | 语义规则实例 |
|------|------------|
| type | `type.dtype = real` |
| decl | `var-list₀.dtype = type.dtype` |
| var-list₀ | `id(x).dtype = var-list₀.dtype` ; `var-list₁.dtype = var-list₀.dtype` |
| var-list₁ | `id(y).dtype = var-list₁.dtype` |

### 4.4 依赖图

```
type.dtype ──→ var-list₀.dtype ──→ id(x).dtype
                               └──→ var-list₁.dtype ──→ id(y).dtype
```

这条链清楚地显示：[[继承属性|继承属性]] `dtype` 由父节点 `.dtype` 出发，**横向**沿兄弟链传递，最终纵向传给每个 `id`。[[L-属性文法|L-属性文法]]的依赖边方向约束在这里得到了精确体现。

## 五、有环依赖与不可计算的属性文法

### 5.1 循环依赖的形式化条件

存在一个属性实例的序列 `a₁, a₂, ..., aₖ, a₁`，使得依赖图中存在边 `a₁ → a₂ → ... → aₖ → a₁`。

### 5.2 典型的循环依赖

```
A.x = B.y + 1
B.y = A.x + 1
```

依赖图：

```
A.x ←→ B.y  （双向边）
```

### 5.3 为什么有环就无法拓扑排序

拓扑排序的定义是：找到一个线性序列，使所有边由前指向后。如果存在环，环上任何节点都不能在序列中排在环上其他节点的「之前」——因为它也同时被闭环上的某个节点所依赖。没有节点可以作为环 of 起点。

### 5.4 有环属性文法的处理

| 策略 | 说明 |
|------|------|
| **不动点迭代** | 给环上属性赋初始猜测值，反复应用语义规则直到收敛（如数据流分析） |
| **文法重设计** | 通过引入中间属性/重构产生式消除环 |
| **拒绝此文法** | 大部分 parser generator 不支持有环的属性文法 |

## 六、拓扑排序算法与属性求值

构造完依赖图后，标准的拓扑排序算法：

```
1. 计算每个节点的入度 indegree
2. 将入度为 0 的节点入队
3. while 队列非空：
     取出队首节点 u，计算 u 对应的属性值
     for u 的每条出边 (u → v)：
        indegree(v)--
        if indegree(v) == 0：v 入队
4. if 已排序节点数 < 总节点数：依赖图存在环 → 报错
```

在 [[S-属性文法|S-属性文法]]中，这个算法退化为简单的**后序遍历**（因为依赖边总是指向祖先方向）。在 [[L-属性文法|L-属性文法]]中，可以通过**深度优先、左→右**的遍历在一次扫描中完成属性和依赖图的计算。

## 七、依赖图在编译理论中的位置

```
CFG + 属性文法
    ↓
语法树 + 语义规则实例
    ↓
构造依赖图
    ↓
拓扑排序（或判定有环）
    ↓
得到属性求值顺序
    ↓
逐一执行语义规则 → 得到完全注解的语法树
```
