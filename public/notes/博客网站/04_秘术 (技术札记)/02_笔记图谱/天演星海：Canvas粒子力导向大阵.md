# 天演星海：Canvas 粒子力导向大阵

这不是一张照搬组件库的普通关系图，而是一座真正由前端手绘出来的"星海推演大阵"。

世俗图谱常以 SVG 显化万千节点：一个节点一个 DOM，一条连线一条 path。小图尚可，一旦 Obsidian 笔记库增长到数百上千篇，节点、连线、标签、hover 高亮全部堆进 DOM 树，浏览器便要同时承担布局、样式计算、事件命中与重绘之苦。星海未成，页面先渡劫。

本阵的破局之法，是将**"力学推演"与"视觉显化"**拆成两层。

---

## 双层的分离

**其一，D3-force 在幕后推演天机。** 节点之间的电荷排斥、连线牵引、速度衰减与空间归位，都由力学引擎持续计算。每一枚知识玉简都像一粒带电星砂，在关系引力中找到自己的位置。

**其二，HTML5 Canvas 在前台一笔显形。** 图谱并不把每个节点都渲染成真实 DOM，而是在单层画布上通过 `paintNode` 与 `paintLink` 绘制圆点、星轨与标签。数千个可视对象最终收束为一张灵画卷，浏览器面对的不再是一片沉重 DOM 森林，而是一块持续更新的像素星盘。

```
graph.json / notes-index.json
        ↓
normalizeGraph / filterGraphByLocal / filterGraphByCollection
        ↓
D3-force 推演节点坐标
        ↓
Canvas 绘制星轨、星砂、标签
        ↓
单层画布承载整片知识星海
```

> D3 管天机，Canvas 管显形。

---

## 节点不是等大的死点

星砂大小并不均匀——节点半径根据连接度生长。关联越多的笔记越醒目，知识网络的重心由数据本身自然浮现，不需要手动标注"这篇很重要"。

标签也不是粗暴全开。它们依据**缩放层级、局部模式、hover 高亮与节点连接度**动态显隐。远看是星云，近看才浮出玉简名。

---

## hover 高亮没有依赖 DOM class

传统做法是 `element.classList.add('highlighted')`，让 CSS 接管样式变化。本阵不这么走。

用高亮集合记录当前节点、邻居节点与相关星轨。下一帧重绘时，Canvas 根据集合调整颜色、透明度与线宽。它不是"修改元素"——是**改变下一笔怎么画**。全程不碰 DOM 树。

---

## 滚轮缩放的阻尼法门

图谱没有让缩放突然跳变。而是维护四组变量：

- 目标缩放、目标中心
- 当前缩放、当前中心

再用 `requestAnimationFrame` 缓慢逼近。星盘被拨动后缓缓归位，用户不会在大型关系网中瞬间迷失方向。手感上不是"看地图"，是**推动一个悬浮在空中的星盘**。

---

## 余量

此阵的意义不止是视觉酷炫。它确保——今日百篇，明日千篇，星海仍可拖拽、缩放、巡游如常。渲染复杂度不随笔记数量线性增长，DOM 负担始终保持恒定：**一张 Canvas。**
