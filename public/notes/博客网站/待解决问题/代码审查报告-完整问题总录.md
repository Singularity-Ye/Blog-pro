## 🔴 一、严重问题 (CRITICAL)

### [C1] `publish-admin.mjs`：路径遍历漏洞 — 任意文件读取

**文件：** `scripts/publish-admin.mjs:167`

```javascript
const full = path.join(manifest.vaultPath, collection.root, ...rel.split('/'));
```

**问题：** `rel` 参数直接来自查询字符串，未做任何校验。攻击者可以通过 `../../` 读取 Vault 之外的任意文件（如 `../../windows/system32/config/sam`），因为代码随后会读取该文件并将内容返回给客户端。

> [!danger] 修复方向
> 在解析 `rel` 之前，先解析并规范化路径，然后验证结果是否仍然以合法的 Vault 目录为前缀。

---

### [C2] `publish-admin.mjs`：管理面板无身份认证

**文件：** `scripts/publish-admin.mjs:1275-1366`

**问题：** 管理服务器在端口 4177 上监听，无任何身份认证。同一网络下的任何设备都可以：
- 修改发布清单
- 触发数据同步
- 读取 Vault 内的任意文件
- 删除分类

> [!danger] 修复方向
> 至少添加基本身份认证，或将其完全限制在 `127.0.0.1` 上监听。该服务器不应在本地开发环境之外运行。

---

### [C3] `Blog.jsx`（4,110 行）：巨型组件

**文件：** `src/pages/Blog.jsx`

**问题：** 这是整个代码库中最大的文件，包含：
- 7 个场景
- Canvas 粒子系统
- SVG 小地图
- 传送动画
- 卷轴面板
- 内联 Mermaid 渲染

全部混在一个文件里，几乎无法调试、测试和维护。

> [!danger] 修复方向
> - 将各场景提取为独立组件
> - 将粒子系统、小地图、传送动画及卷轴面板分离为各自的组件
> - 预计工作量：4-6h

## 🟠 二、高危问题 (HIGH)

### [H1] Mermaid SVG 渲染使用 `dangerouslySetInnerHTML`

**文件：** `Note.jsx:748` / `Blog.jsx:2859`

两个文件都使用 `dangerouslySetInnerHTML` 渲染 Mermaid 图表。虽然 Mermaid 会做一定程度的清理，但这仍是运行用户控制内容的攻击面。

### [H2] Mermaid `securityLevel: 'loose'`

**文件：** `Note.jsx:685` / `Blog.jsx:2796`

`securityLevel: 'loose'` 设置允许用户在流程图中执行任意脚本。鉴于内容来自自己的 Obsidian Vault，风险可控，但仍需注意。

### [H3] `publish-admin.mjs`：内联 HTML SPA（~1,100 行）

**文件：** `scripts/publish-admin.mjs:1098`

整个管理控制台 UI（HTML + CSS + JS）全都在一段 JavaScript 模板字符串字面量中，样式调整、调试和维护都极其痛苦。

> [!warning] 修复方向
> 拆分为独立的前端项目或至少独立 HTML/CSS/JS 文件。预计工作量：4h。

### [H4] `Lanyard.jsx`（~980 行）：巨型 3D 组件

**文件：** `src/components/Lanyard/Lanyard.jsx`

将物理模拟（Rapier rope joints）、纹理合成（Canvas 2D 绘制头像+卡片）、动画以及交互逻辑全部混在一个文件里。

### [M3] 缺少懒加载 / 代码拆分

所有页面及 3D 库均在首屏加载。Three.js（~600KB gzip）、D3.js、Mermaid 和 Matter.js 全部打包进主 bundle。`React.lazy` + `Suspense` 可以极大改善初始加载性能。

### [M5] `graph.json` 被多次重复请求

`graph.json` 在 `Note.jsx`、`Atlas.jsx` 和 `GraphView.jsx` 中各自独立拉取，且无缓存。前端路由切换时可能导致 3 次完全相同的网络请求。

> [!tip] 修复方向
> 使用 React Context 或简单的模块级缓存，复用同一份图谱数据。
