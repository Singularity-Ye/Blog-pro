# Codex 三件套：AI 做 PPT 的最强组合

> [!abstract] 一句话
> 用 Codex 做总导演，Image Generation Skill 出视觉素材，Presentation 插件排版导出——三个工具串成一条管线，最终产出**可编辑的 PPTX 文件**。中间设了人工检查点，不是一键到底，而是在关键节点让你拍板。

---

## 一、三件套分工

| 角色 | 工具 | 干什么 |
|------|------|------|
| **总导演** | Codex | 理解需求 → 拆解任务 → 决定什么时候用图、什么时候排版 → 管理全流程 |
| **美术** | Image Generation Skill | 生成所有视觉素材：背景、装饰元素、图表雏形。输出 PNG |
| **排版师** | Presentation 插件 | 把 Image 生成的 PNG 素材排到页面上，加文字，调版式，导出可编辑的 PPTX |

三者的关系是**串行但有反馈**——Codex 不直接替你做完所有事，它在每个关键节点停下来让你看一眼，确认了再继续。

---

## 二、完整工作流

### 第一步：让 Codex 出大纲，先不动手

```text
请按照这个内容做一个八页左右的 PPT。
你先设计每页的大纲和内容，给我预览。
等我确认之后再进行下一步。
```

为什么不让它直接做？因为一键到底出来的东西，大概率不是你想要的。中间不设检查点 = 赌运气。

Codex 这一步输出：每页标题 + 要点 + 建议的视觉方向。

### 第二步：浓缩预览——这个技巧很聪明

确认大纲后，不急着逐页出图。而是：

```text
调用 Image Generation，给我三套不同的视觉风格。
每套把全部八页浓缩在一张图里。我选一个再继续。
```

**这一步是整个流程的精髓。**

传统做法是"生成第一页 → 不满意 → 重新生成 → 又不满意 → 再来"，试错成本高。他把八页缩成一张风格预览图，一次性对比三种风格，选完了再往下走。Image 模型完全有能力把多页信息浓缩到一张图里——这不是 bug，是技巧。

### 第三步：确认风格，拆分生成素材

选定风格后，两条路：

**路径 A（快速版）**：直接让 Image 把八页按图片格式输出 → 拼成 PDF。快，但不可编辑。适合"看一眼就行"的场合。

**路径 B（专业版）**：让 Image 把页面里的**每一个视觉元素**单独生成为 PNG，然后交给 Presentation 插件：

```text
按第三种风格生成 PPT。
第一，把里面的每个视觉元素、文字都按 PNG 格式生成。
第二，用 Presentation 插件排版，导出为 PPTX 文档。
```

这一步耗时最长（演示里跑了约十分钟）。但最终的产物是一个**真正可编辑的 PPTX 文件**——每个元素可以拖、可以改、可以调。

## 三、和 [[PPT制作新思路：AI生图做皮肤，代码画布做骨架|HTML/CSS/Canvas 方案]] 的对比

两套方案讲的是同一件事——"AI 出皮肤，精确工具做骨架"——但路径不同：

| 维度 | HTML/CSS/Canvas 方案 | Codex 三件套 |
|------|------|------|
| **精确排版靠什么** | 你手写 HTML/CSS，代码控制一切 | Presentation 插件，AI 自动排版 |
| **视觉素材来源** | Midjourney / DALL-E 出背景图 | Codex 内置 Image Generation Skill |
| **人工介入点** | 全程手动——写代码、调 CSS、导出 | 三个检查点：大纲确认 → 风格选择 → 最终验收 |
| **输出格式** | HTML → 浏览器打印 PDF / Puppeteer 截图 | 可编辑的 PPTX 文件 |
| **修改方便度** | 改代码 → 重新渲染 → 重新导出 | 直接在 PowerPoint 里拖拽编辑 |
| **适合谁** | 会写代码、追求像素级控制 | 不想写代码、需要可编辑的最终文件 |
| **核心优势** | 精确到像素，数据驱动（图表可以自动更新） | 流程顺滑，产出是通用格式，非技术人员也能接手 |

### 什么时候用哪个？

```
你需要图表和数据绑定？        → HTML/CSS/Canvas 方案
你需要交付给同事继续编辑？    → Codex 三件套
你追求极致视觉控制？         → HTML/CSS/Canvas 方案
你想快速出初稿再人工精修？    → Codex 三件套
你做的是技术分享（代码块多）？ → HTML/CSS/Canvas 方案（Slidev）
你做的是商业汇报？            → Codex 三件套
```

两套方案不是替代关系，是**互补**。甚至可以混用——用 Codex 出视觉素材和版式框架，导出 PPTX 后再手动用 Canvas 补数据图表。

## 四、这套方案的三个亮点

### 1. 浓缩预览法

不是一页一页试风格。把八页浓缩到一张图，一次对比三种风格。**在代价最低的时候做决策。** 这条原则可以推广到任何 AI 生成任务——先生成缩略图确认方向，再展开细节。

### 2. 人工检查点设在关键分叉口

不是"一键到底"，也不是"每步都审"。在大纲确认、风格选择这两个分叉口设检查点。路径定了之后，让 AI 跑完剩下的。**检查点的密度刚好——太少容易翻车，太多等于自己干。**

### 3. 素材和排版分离

Image 只出素材 PNG，Presentation 只做排版。两个工具各干各的，互不干扰。如果排版不满意，换素材就行；如果素材不满意，重新生成素材就行。不用因为一个元素不对而重做整页。

这也恰好印证了 [[Obsidian本质理解：Markdown、HTML 与 AI 时代的知识工作流|Obsidian 本质理解]] 里那个底层哲学——**分层解耦，让每层专注于自己最擅长的事。**
