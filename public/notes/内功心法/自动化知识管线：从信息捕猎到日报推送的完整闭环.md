# 自动化知识管线：从信息捕猎到日报推送的完整闭环

> [!abstract] 一句话
> 用 Hermes 做调度中枢，AutoCLI 做信息捕猎，LLM Wiki Skill 做知识编译，微信 Clawbot 做日报推送——把"刷信息 → 存笔记 → 回顾"这条链路全部自动化。知识库不再是一个仓库，而是一台每天为你工作的机器。

---

## 一、三阶段管线

整个流程拆成三段，每段由一个组合负责：


## 二、各组件职责

### Hermes —— 调度中枢

Hermes 是整条管线的"大脑"。它的角色不是干活的，是**指挥别人干活**：

- 定时触发：每天下午 5 点自动执行全流程
- 串联步骤：阶段一完成 → 自动启动阶段二 → 完成后启动阶段三
- 容错：某个源抓失败了不影响其他源继续跑
- 状态汇报：每一步的执行结果可追溯

没有 Hermes 的话，这三个步骤就是三个孤立的脚本，需要人手动串起来。

### AutoCLI —— 信息捕猎手

AutoCLI 负责从各种信息源抓取原始内容。它有几个关键设计：

- **复用 Chrome 登录态**——不需要单独申请 API Key，直接用你浏览器里已经登录的账号去抓。这意味着可以抓 X（Twitter）的热门推文、微信公众号文章、付费订阅内容——这些用 API 要么贵、要么不开放、要么需要企业认证
- **支持 90+ 站点**——覆盖面够广，不是只能抓一两个平台
- **输出原始文章**——不做处理，原样交给下一阶段的 LLM 去理解和整理

这本质上是一个**合法登录态下的网页内容提取器**，区别于传统的"爬虫"。它抓的是"你本来就订阅了、关注了、有权看到的内容"，只是帮你去掉了手动刷的步骤。

### LLM Wiki Skill —— 知识编译引擎

这是最关键的一步——把**原始信息流压成结构化知识**。

原始文章是一篇一篇的，来源混杂、格式不一、质量参差。LLM 做的事情是：

1. **去噪**——过滤掉广告、评论区垃圾、重复转载
2. **总结**——每篇提炼出核心观点、关键数据、可操作的结论
3. **结构化**——按主题分类，打标签，建立 Wiki-link
4. **关联**——新内容和知识库里已有内容建立链接
5. **入库**——写入 Obsidian Vault 的指定目录，按日期归档

产出的不是"今天收藏了 10 篇文章"，而是"今天有 10 个新知识点被编译进了你的知识体系，其中 3 个和已有的内容有关联"。

这就是从"信息收藏夹"到"可生长的知识库"的质变。

### 微信 Clawbot —— 日报推送

入库完成后，Hermes 触发日报生成，通过微信发到手机上。日报内容：

- **今日总览一句话**——今天知识库里发生了什么
- **新增数量**——几篇文章入库
- **今日重点**——哪篇最值得看
- **100 字摘要**——重点文章的提炼
- **报告位置**——在 Obsidian 的哪个目录下

注意：日报不是"今天互联网上发生了什么"，而是"今天你的知识库里新增了什么"。视角从"外部信息流"转向了"内部知识增长"。

## 三、这套方案解决了什么问题

传统的信息管理方式是这样的：

```
刷微博/推特/公众号 → 觉得有用 → 收藏/书签/转发到文件传输助手 → 再也不看
```

每个环节都在漏：

| 环节 | 漏了什么 |
|------|------|
| **发现** | 刷到什么是算法决定的，不是你的知识需求决定的 |
| **收藏** | 散落在不同平台——微博收藏、微信收藏、浏览器书签、截图相册 |
| **整理** | 永远不会整理。收藏即遗忘 |
| **回顾** | 没有回顾机制。收藏夹是一座坟 |

这套管线把四个环节全部自动化改造了：

| 旧方式 | 新方式 |
|------|------|
| 手动刷 → 被动接收算法推荐 | AutoCLI 定时抓取 → 按配置好的信息源主动捕猎 |
| 散落各处 → 跨平台碎片 | 统一存入 Obsidian → 单一知识库 |
| 永远不整理 → 收藏即遗忘 | LLM 自动编译 → 结构化 + 标签 + 关联 |
| 无回顾 → 收藏夹是坟 | 微信日报主动推送 → 每天告诉你知识库长了什么 |

## 四、和我的 Obsidian 知识体系的对比

我现在的工作流是"手动驱动"的：

```
我想写东西 → 查资料 → 写 Markdown → Claudian 整理 → Obsidian 渲染 → 博客展示
```

这套管线是"自动驱动"的：

```
AutoCLI 抓取 → LLM 编译 → 自动入库 Obsidian → 微信日报通知
                                           ↓
                              人类（我）在有空的时候去看、去关联、去深化
```

两条线其实可以交汇。

自动管线负责**"信息捕猎和粗加工"**——喂进来的是一堆原始文章，生成的是结构化但还比较浅的 Wiki 条目。然后人类 + Claudian 负责**"深度加工"**——挑选有价值的条目，补充自己的思考，建立和已有笔记的深度链接，写成方法论级别的长文。

就像采矿业：
- AutoCLI = 挖掘机，把矿石挖出来
- LLM = 初选厂，把矿石粉碎、分拣、粗炼
- 我 + Claudian = 精炼厂，把粗矿炼成金属、做成零件、搭成机器

## 五、潜在问题和注意点

### 信息过载的反面：信息噪音

自动抓取 + 自动入库的一个风险是——**量太大了。** 每天 10 篇，一个月 300 篇，一年 3600 篇。如果 LLM 编译的质量不够高，知识库会变成另一个收藏夹——只不过这次是"自动填满的"。

解决办法：**日报不是让你每篇都看。日报是让你知道"有什么"，你挑 1-2 篇真正值得深读的去读。** 剩下的就留在知识库里当存档，以后搜索时能命中就行。

### 登录态的安全问题

AutoCLI 复用 Chrome 登录态——这意味着它有权限访问你登录状态下的所有内容。如果 AutoCLI 本身有安全漏洞，或者被供应链攻击污染，你的账号信息可能泄露。

这和之前 [[9块9的代价：你的AI中转站正在帮你装木马|那篇 9 块 9 中转站]] 讨论的问题类似——信任链条长了，攻击面就大了。

### 对 Obsidian Vault 的"入侵"

自动入库意味着机器人会在你的知识库里创建文件。如果你的 vault 结构是精心手打的，突然出现一个 `AI NEWS HUB/` 目录，里面每天自动新增 10 个文件——这到底是"知识增长"还是"知识污染"？

建议：**给自动入库的内容单独划一块区域**（比如 `_inbox/` 或 `AI NEWS HUB/`），和手动写的笔记物理隔离。只有当你亲手把某篇从"自动收录区"移到"个人知识区"时，它才真正成为你知识体系的一部分。

## 六、如果要自己搭，最小可行方案

不一定需要 Hermes + AutoCLI 这整套。拆开看核心能力：

| 能力 | 轻量替代 |
|------|------|
| 定时触发 | Windows 任务计划 / cron / GitHub Actions |
| 网页抓取 | 浏览器扩展 MarkDownload / SingleFile + 手动保存，或者用 Python `requests` + `beautifulsoup` |
| LLM 总结 | Claudian 批量读取 + 总结（你已经在用了） |
| 自动入库 | Claudian 写入 Obsidian vault（已经在做了） |
| 微信通知 | Server 酱 / PushPlus / 企业微信机器人 Webhook |

最小可行版本：

```
每天下午 5 点 → 脚本抓取关注的信息源 → 存为 Markdown 到 _inbox/
→ Claudian 批量总结 + 打标签 → 微信发一条"今日入库 N 篇，重点是 X"
```

---

> [!summary]
> 这套方案的真正价值不是"自动化"本身，而是把知识管理从**被动消费**变成了**主动经营**。
> 你不是在刷信息，你是在经营一座知识矿场。挖矿自动化了，勘探和精炼才是你该花时间的地方。
