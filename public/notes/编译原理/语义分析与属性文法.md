# 语义分析与属性文法

> 语法正确≠语义正确。属性文法为 CFG 的每个符号和产生式附加语义规则，综合属性自底向上传递值，继承属性自顶向下/自左向右传递上下文。

---

## 一、[[编译原理/关系图谱-节点内容/静态语义|静态语义]]分析的形式化定位

| 阶段 | 检查的问题 | 输出 |
|------|-----------|------|
| 词法分析 | 字符能拼成合法的 token 吗？ | token 流 |
| 语法分析 | token 序列符合 CFG 吗？ | 语法树 |
| **语义分析** | 这个程序合法吗？（类型、声明、作用域） | 带属性的语法树 |
| 中间代码生成 | — | 三地址码等 |

### 语义分析的具体检查项（形式化）

| 检查类别 | 形式化判据 | 示例 |
|----------|-----------|------|
| **声明检查** | ∀ id ∈ program, ∃ declaration(id) ∧ declaration 在 id 的使用之前 | `x = 3;` 但 `x` 未声明 |
| **类型相容性** | `expr.left.type` 与 `expr.right.type` 在运算符 `op` 下可合式 (well-typed) | `"hello" + 3` |
| **赋值相容性** | `expr.type ≤ var.type`（类型可赋值到变量类型） | `int x = 3.14;` |
| **函数调用** | 实参数量=形参数量 ∧ ∀i: `argᵢ.type ≤ paramᵢ.type` | `f(1, "a")` 但 `f(int, int)` |
| **作用域** | 每个名字的引用能解析到**唯一**的声明实例 | 内层变量遮盖外层同名变量 |

---

## 二、静态语义 vs 动态语义

| 类型 | 检查时机 | 典型例子 |
|------|----------|----------|
| **静态语义** | 编译时（不运行程序） | `int x = "hello";` —— 类型不匹配 |
| **动态语义** | 运行时（程序执行期间） | `a / b` —— `b == 0` 引发除零错误 |
| **不可判定的属性** | 编译器无法静态决定 | `malloc` 是否成功、数组越界（一般需运行时检查） |

语义分析只覆盖**静态可判定的语义规则**。

## 三、属性 (Attribute)：形式化定义

### 3.1 定义

设一个 CFG 的文法符号（终结符或非终结符）为 X。**属性** a 是一个映射：

```
a: parse-tree-node-of-X → 值域
```

对于语法树的每个 X 节点，`X.a` 取该值域中的一个具体值。

### 3.2 两类属性

| 类型 | 英文 | 形式化定义 | 典型值域 |
|------|------|-----------|------|
| **综合属性** | Synthesized attribute | 由子节点的属性计算得到 | `val ∈ ℤ`, `type ∈ {int, real, bool}`, `code ∈ 三地址码*` |
| **继承属性** | Inherited attribute | 由父节点或兄弟节点的属性计算得到 | `dtype ∈ {integer, real}`, `scope ∈ 符号表`, `base ∈ {8, 10}` |

### 3.3 常用属性一览

| 属性 | 类型 | 取值示例 | 所属符号 |
|------|:--:|------|------|
| `val` | 综合 | `7`, `1302` | `number`, `exp`, `term`, `factor`, `digit` |
| `type` | 综合 | `integer`, `real`, `error` | `exp`, `term` |
| `dtype` | 继承 | `integer`, `real` | `id`, `var-list` |
| `base` | 继承 | `8`, `10` | `digit`, `num` |
| `name` | 综合 | `"x"`, `"foo"` | `id` |
| `code` | 综合 | 三地址码序列 | `stmt`, `exp` |
| `addr` | 综合/继承 | 内存地址 | `id`, `exp`（代码生成阶段） |

## 四、语义规则与[[编译原理/关系图谱-节点内容/属性文法|属性文法]]

### 4.1 语义规则的形式化定义

对每条产生式 `p: X₀ → X₁ X₂ ... Xₙ`，语义规则是形如以下两种的方程：

```
综合属性：X₀.a = f( X₁.b, X₂.c, ... , Xₙ.m )     ← [[编译原理/关系图谱-节点内容/综合属性|综合属性]]
继承属性：Xᵢ.b = f( X₀.a, X₁.c, ... , Xᵢ₋₁.m )     ← [[编译原理/关系图谱-节点内容/继承属性|继承属性]]
```

### 4.2 属性文法的完整定义

一个 **[[编译原理/关系图谱-节点内容/属性文法|属性文法]] (Attribute Grammar, AG)** 是一个三元组 `AG = (G, A, R)`，详细定义见该节点。

### 4.3 例题一：无符号整数的值计算（纯综合属性）

**文法**：
```
number → number digit | digit
digit → 0 | 1 | ... | 9
```

**属性**：`val`（综合）——每个非终结符和终结符都有。

**语义规则**：

| 产生式 | 语义规则 |
|--------|----------|
| `number₁ → number₂ digit` | `number₁.val = number₂.val × 10 + digit.val` |
| `number → digit` | `number.val = digit.val` |
| `digit → 0` | `digit.val = 0` |
| ... | ... |
| `digit → 9` | `digit.val = 9` |

**推导 345 的属性计算过程**：

```
          number (val=345)
         /        \
    number (val=34)  digit (val=5)
    /        \
number (val=3)  digit (val=4)
    |
  digit (val=3)
```

全部综合属性，自底向上一次遍历可以算出所有值。

> 详见 [[编译原理/依赖图与属性计算#三、例子：数字求值|依赖图对数字求值的拓扑分析]]。

### 4.4 例题二：算术表达式求值（纯综合属性，S-属性文法）

**文法**：
```
exp    → exp + term | exp - term | term
term   → term * factor | factor
factor → ( exp ) | number
```

**属性**：`val`（综合）。

**语义规则**：

| 产生式 | 语义规则 |
|--------|----------|
| `exp₁ → exp₂ + term` | `exp₁.val = exp₂.val + term.val` |
| `exp₁ → exp₂ - term` | `exp₁.val = exp₂.val - term.val` |
| `exp → term` | `exp.val = term.val` |
| `term₁ → term₂ * factor` | `term₁.val = term₂.val * factor.val` |
| `term → factor` | `term.val = factor.val` |
| `factor → ( exp )` | `factor.val = exp.val` |
| `factor → number` | `factor.val = number.val` |

**计算 `(34 - 3) * 42`**：

```
factor(34) → number(34).val = 34
factor(3) → number(3).val = 3
exp(31) → exp(34).val - term(3).val = 31
factor(31) → exp(31).val = 31
factor(42) → number(42).val = 42
term(31 × 42) → term(31).val × factor(42).val = 1302
```

全部是综合属性，分析树自底向上求值。这称为 **[[编译原理/关系图谱-节点内容/S-属性文法|S-属性文法]]**——只有[[编译原理/关系图谱-节点内容/综合属性|综合属性]]，没有[[编译原理/关系图谱-节点内容/继承属性|继承属性]]。S-属性文法可以在 LR 自底向上分析的同时完成属性求值。

### 4.5 例题三：变量声明的类型传播（综合 + 继承）

**文法**：
```
decl → type var-list
type → int | float
var-list₁ → id , var-list₂ | id
```

**属性**：
- `dtype`（继承）——从 type 流向每个 id
- `type.dtype` 实际上可以看作综合属性（从子节点 int/float 传上来的具体类型值）

**语义规则**：

| 产生式 | 语义规则 |
|--------|----------|
| `decl → type var-list` | `var-list.dtype = type.dtype` |
| `type → int` | `type.dtype = integer` |
| `type → float` | `type.dtype = real` |
| `var-list₁ → id , var-list₂` | `id.dtype = var-list₁.dtype` ; `var-list₂.dtype = var-list₁.dtype` |
| `var-list → id` | `id.dtype = var-list.dtype` |

**类型传播图 (`float x, y`)**：

```
type (dtype=real)
    ↓ 继承
var-list (dtype=real)
    ↓ 继承            ↓ 继承
id(x).dtype=real   id(y).dtype=real
```

继承属性 `dtype` 由父节点 `var-list` 传给子节点 `id`。这种包含[[编译原理/关系图谱-节点内容/继承属性|继承属性]]的文法称为 **[[编译原理/关系图谱-节点内容/L-属性文法|L-属性文法]]**——属性的依赖边只沿左兄弟和父节点到子节点方向流动，适合在深度优先遍历中计算。

---

## 五、S-属性 vs L-属性的形式化区分

| 文法类别 | 属性类型 | 求值策略 | 适用场景 |
|----------|:--:|------|------|
| **S-属性文法** | 仅有综合属性 | LR 自底向上分析时同步求值 | 表达式求值、数值计算 |
| **L-属性文法** | 综合 + 继承（左→右，父→子） | 深度优先遍历（左→右） | 声明类型传播、符号表构建 |

所有 S-属性文法也是 L-属性文法（继承属性集合为空的特殊情况）。

## 六、语法合法 ≠ 语义合法

```
文法允许 189o（digit 可以是 0-9）
但 semantic rule 要求八进制 digit ∈ [0, 7]
```

语义规则必须在值域上做约束：

```
digit → 8
  digit.val = if base == 8 then ERROR else 8

digit → 9
  digit.val = if base == 8 then ERROR else 9
```

**核心教训**：CFG 只管句法形式，语义规则负责内容的合法性。语法正确只是必要条件，语义检查才是编译器给程序"盖章"的最后一步。
