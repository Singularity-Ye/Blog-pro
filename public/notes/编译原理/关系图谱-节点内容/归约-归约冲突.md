# 归约-归约冲突 (Reduce-Reduce Conflict)

**归约-归约冲突**是指分析器在当前状态下，发现可以同时使用两条不同的文法规则进行[[归约]]，从而产生歧义。这种冲突通常比 [[移进-归约冲突]] 更严重，因为无法通过优先级声明来简单解决。

## 产生原因

当一个 [[DFA]] 状态中同时包含两个或以上的"完整项目"时：
- $A \to \alpha \cdot$（想归约为 $A$）
- $B \to \beta \cdot$（想归约为 $B$）

分析器无法确定到底应该按哪条规则进行归约。

### 具体例子

```c
// 典型的归约-归约冲突场景
stat → id  (;    // 赋值语句 id = expr;
stat → id  (    // 函数调用 func();
```

假设栈顶已匹配到 `id`，两个规则都处于"刚好完整"的状态：
```
stat → id ·    （规则 1：赋值语句的前半部分）
stat → id ·    （规则 2：函数调用的前半部分）
```

如果下一个输入符号在两者的 [[Follow 集合]] 中都有，SLR(1) 也无法区分。

### 更现实的例子：悬挂 else

在很多语言的文法设计中，"if 语句"和"if-else 语句"容易引发归约-归约冲突：

```
stmt → if e then stmt           （规则 1）
stmt → if e then stmt else stmt （规则 2）
```

解析到 `if e then stmt` 时，既可以用规则 1 归约，也可以继续移进 `else`。

## 解决办法

### 方法一：SLR(1) —— Follow 集合区分

与 [[移进-归约冲突]] 类似，[[SLR(1)]] 会通过查看当前输入字符属于 $A$ 还是 $B$ 的 [[Follow 集合]] 来决定用哪个规则归约。

如果 $Follow(A)$ 与 $Follow(B)$ **无交集**，则 SLR(1) 可以完美解决。但如果输入字符同时存在于两者的 Follow 集合中，则 SLR(1) 也无法解决，需要更高级的 LR(1) 算法。

### 方法二：文法重构（最根本的解决方式）

归约-归约冲突往往暗示**文法本身存在歧义或设计缺陷**。最好的办法是重构文法，将冲突消除在源头上。

例如上面的"悬挂 else"问题，可以通过将语句分为"完整语句"和"未封闭语句"两个层次来解决。

### 方法三：使用更强大的分析算法

- **LR(1)**：通过更精确的向前看符号集合，能区分大部分归约-归约冲突
- **LALR(1)**：在 LR(1) 基础上合并等价状态，兼顾冲突消解与表大小

## 移进-归约 vs 归约-归约冲突对比

| 维度 | 移进-归约冲突 | 归约-归约冲突 |
|------|-------------|-------------|
| 冲突性质 | 动作方向（吃 vs 吐）冲突 | 规则选择（用哪条）冲突 |
| 严重程度 | 较轻 | 较重 |
| 优先级声明 | 通常可以解决 | 无法解决 |
| 文法重构 | 可选优化 | 通常必须 |
| LR(1) 解决能力 | 几乎全部解决 | 大部分解决 |
