# LALR(1) 分析算法

**LALR(1)**（Look-Ahead LR）是 LR(1) 的工程压缩版。它把 LR(1) 中核心相同、仅 lookahead 不同的状态合并为一个，从而将状态数缩到 LR(0) 级别，同时保留接近 LR(1) 的识别能力。**Yacc / Bison 使用的就是 LALR(1)**。

## 为什么要压缩 LR(1)

LR(1) 的状态数是其最大的工程障碍。同一个文法在 LR(0) 下可能有 n 个状态，在 LR(1) 下膨胀到 5n 到 15n。LALR(1) 的核心观察是：

> 许多 LR(1) 状态膨胀的原因仅仅是 lookahead 不同——去掉 lookahead 后，它们是一模一样的 LR(0) 核心。

## 合并的形式化定义

### 状态的核心

对于一个 LR(1) 状态 I，其**核心** `core(I)` 定义为去掉所有 lookahead 后的 LR(0) 项目集：

```
core(I) = { A → α·β | [A → α·β, a] ∈ I }
```

### 合并算法

```
输入：LR(1) 项目集规范族 C_LR1 = {I₁, I₂, ..., Iₙ}
输出：LALR(1) 项目集规范族 C_LALR1

1. For LR(1) 规范族中的每个状态 I：
     计算 core(I)
2. 将 core 相同的状态合并为一个状态 K：
     K = { [A → α·β, a] | 存在 LR(1) 状态 I, core(I) = core(K)
                          且 [A → α·β, a] ∈ I }
     即：LR(0) 核心相同 → 合并，lookahead 取所有来源的并集
3. 合并后的状态集合即 LALR(1) 规范族
```

### 合并对分析表的影响

| 表类型 | 影响 |
|--------|------|
| GOTO 表 | 核心相同 → GOTO 关系保持一致（因为 GOTO 不依赖 lookahead） |
| ACTION shift | 完全保留——移进不依赖 lookahead |
| ACTION reduce | lookahead 是多个源状态的并集 → 比单一 LR(1) 状态**宽松** |
| **潜在新增冲突** | 合并后 lookahead 取并 → 原本在 LR(1) 下分别安全的两个归约，合并后可能产生 **reduce/reduce 冲突** |

## LALR(1) 不会引入新的 shift/reduce 冲突

一个重要的理论性质：LALR(1) **不会引入 LR(1) 原本没有的 shift/reduce 冲突**。

证明思路：shift/reduce 冲突来自「同一个状态下、对同一个输入符号 a，同时有移进项目和归约项目」。如果合并后的 LALR(1) 状态 K 对 a 同时有 shift 和 reduce，那么 merge 前必定存在某个 LR(1) 源状态 I 也对 a 有同样的冲突（因为 shift 和 reduce 的 lookahead 都是从源状态继承的）。LR(1) 没有此冲突 → LALR(1) 也不会有。

> 但 LALR(1) **可能引入** LR(1) 没有的 **reduce/reduce 冲突** ——因为两个源状态分别的归约对不同的 lookahead 安全，合并后 lookahead 取并，重叠。

## 能力层级

```
LR(0) ⊂ SLR(1) ⊂ LALR(1) ⊂ LR(1)
  ↑       ↑         ↑         ↑
最小   加全局    合并LR(1)  每个项目
        FOLLOW   核心相同   自带lookahead
```

- 任何 LALR(1) 文法也是 LR(1) 文法
- LALR(1) 能处理的文法是 SLR(1) 的超集
- 实际编程语言文法绝大多数可写成 LALR(1)

## GOTO 表压缩的副作用

由于合并了核心相同的状态，LALR(1) 的 GOTO 函数从 LR(1) 层面缩到了 LR(0) 层面。这意味着 LALR(1) 在**错误检测**上可能比 LR(1) 略差：LR(1) 有更多状态，能在更精确的上下文位置发现语法错误；LALR(1) 合并后可能多做几次归约才报错。但这对实际影响很小——错误最终仍会被发现。

## LALR(1) 与 Yacc 的关系

Yacc 内部构造 LALR(1) 分析表的方法并非"先构造完整 LR(1) 规范族再合并"——那太浪费中间状态。实际算法是更高效的增量式构造，但**最终结果等价**。
