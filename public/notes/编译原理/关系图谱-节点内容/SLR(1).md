# SLR(1) (Simple LR)

**SLR(1)** 是一种简单但比 LR(0) 聪明的[[自底向上语法分析]]算法。它在 LR(0) 的基础上引入了"向前看一个符号"的能力，以极小的代价化解了大量 [[移进-归约冲突]]。

## 核心区别

- **S** 代表 Simple（简单）——算法简单，表体积与 LR(0) 相同。
- **1** 代表 Lookahead 1 Token（向前看 1 个字符）。
- 它直接借用了 LR(0) 粗糙的 [[DFA]] 状态图，但是通过引入 [[Follow 集合]] 进行拦截，解决了 LR(0) 盲目归约造成的问题。

## SLR(1) 归约决策规则

```
在状态 I 下，遇到输入字符 a：
  如果有完整项目 A → α· 且 a ∈ Follow(A) → 执行归约
  否则如果有移进项目 A → α·aβ        → 执行移进
  否则如果有完整项目但 a ∉ Follow(A) → 不归约（解决冲突！）
```

### 对比 LR(0) 与 SLR(1)

| 场景 | LR(0) | SLR(1) |
|------|-------|--------|
| 状态 $I$ 有 $T \to T \cdot * F$ 和 $F \to id \cdot$ | 遇到 `*` 时冲突 | 如果 $* \notin Follow(F)$ → 只移进，不归约 |
| 状态 $I$ 有 $E \to T \cdot$ 时遇到 `$`（结束符） | 直接归约 | 如果 `$` $\in Follow(E)$ 才归约 |
| 表大小 | 小 | 与 LR(0) **完全相同** |
| 冲突数量 | 多 | 中等 |

## SLR(1) 的局限性

虽然叫 SLR(1)，但它有个重要缺陷：**Follow 集合过于粗糙**。

### 问题说明

```
A → α    其中 Follow(A) = {a, b}
B → β    其中 Follow(B) = {b, c}
```

如果 DFA 的某个状态同时包含 $A \to \alpha \cdot$ 和 $B \to \beta \cdot$，且输入符号是 $b$（同时属于两者的 Follow 集合），则 SLR(1) 仍然无法决定用哪条规则归约——这就是 [[归约-归约冲突]]。

### 示例：SLR(1) 无法解决的经典场景

```
S → id = E
S → id : T
```

两个产生式都产生 $S$，且 $Follow(S)$ 重叠时，SLR(1) 无法区分 `id = ...` 和 `id : ...`。

## 不同 LR 算法对比

| 算法 | 构建复杂度 | 状态数 | 冲突消解能力 | 适用场景 |
|------|-----------|--------|-------------|---------|
| **LR(0)** | 低 | 较少 | 几乎不消解 | 教学/极简单的文法 |
| **SLR(1)** | 中 | 同 LR(0) | 消解部分，靠 Follow | 多数编程语言的表达式 |
| **LALR(1)** | 中高 | 同 LR(0) | 消解大部分冲突 | YACC/Bison 默认算法 |
| **LR(1)** | 高 | 可能膨胀数倍 | 几乎消解所有冲突 | 需要高精度的语法分析 |

## 何时升级到更强的算法？

如果 SLR(1) 报告[[归约-归约冲突]]，并且通过文法重构无法解决，就需要升级到更强（也更复杂）的 **LALR(1)** 或 **LR(1)** 算法。LALR(1) 继承 SLR(1) 的状态数量优势，但通过更精确的向前看信息来消解冲突。
