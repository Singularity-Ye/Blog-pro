# 核心项 (Kernel Items)

在一个 [[DFA]] 状态内部，**核心项**是这个状态的"老大哥"和"种子"——它是状态中最核心、最不可替代的项目。

## 定义

除了 [[增广文法]] 的初始项目（$S' \to \cdot S$）之外，所有**圆点（·）不在最左边**的 [[LR(0) 项目]] 都是核心项（例如：$A \to X \cdot Y Z$）。

更精确地说，一个 DFA 状态 $I$ 中的核心项是：
1. 从上一状态通过 [[跳转函数]] $GO(I_{prev}, X)$ **移点进入**的项目（圆点不在最左边）。
2. 初始状态中的 $S' \to \cdot S$（虽然圆点在最左边，但它是特殊的"起点"）。

### 特征判断

```
E' → ·E         ← 核心项（规则 2：初始状态的起点）
E  → E·+ T     ← 核心项（规则 1：圆点不在最左边）
E  → ·E+T      ← 闭包项（圆点在最左边，非初始项目）
T  → ·id       ← 闭包项（圆点在最左边）
```

## 为什么需要"核心项"这个概念？

### 编译器实现中的内存优化

在编译器底层的 LR 表生成器中，每个 DFA 状态通常只存储核心项，因为：

- **核心项是"种子"**：给定一个核心项集合，通过 [[闭包运算]] 可以完全恢复出整个状态（核心项 + 所有 [[闭包项]]）。
- **闭包项可推导**：闭包项是核心项的"衍生物"，不需要持久化存储。

```
状态存储优化
┌─────────────────────────┐
│  完整状态（占用 100%）   │
│  ┌───────┐              │
│  │ 核心项 │ ← 只存这些  │
│  │ (25%)  │   就够了     │
│  └───────┘              │
│  ┌─────────────┐        │
│  │  闭包项(75%)  │       │
│  │  (必要时展开)  │       │
│  └─────────────┘        │
└─────────────────────────┘
```

## 核心项 vs 闭包项

| 维度 | 核心项 | [[闭包项]] |
|------|--------|-----------|
| 圆点位置 | 不在最左边（或初始 $S' \to \cdot S$） | 在最左边 |
| 来源 | 移点进入 / 初始 | [[闭包运算]] 展开 |
| 存储必要性 | 必须存储 | 可实时推导 |
| 代表含义 | 已匹配至少一个符号的进度 | 即将可能匹配的符号 |
| 数量占比 | 较少（约 25%） | 较多（约 75%） |

## 示例：LR 状态中的核心项

考虑状态 $I_2$（通过 $GO(I_0, T)$ 到达）：

```
从 I₀ 经过 T 跳转进入 I₂：
进入方式：从 I₀ 中 E → ·T，圆点跳过 T → E → T·

I₂ 的核心项：
  E → T·                  ← 圆点不在最左边（唯一核心项）

I₂ 的完整状态（核心项 + 闭包运算展开）：
  E → T·                  ← 核心项
  （圆点后无符号，无闭包展开）
```

核心项是状态机中最精炼的**记忆载体**——每个核心项都代表了一条从初始状态到当前状态的**推导路径**。
