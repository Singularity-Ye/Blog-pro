# LR(0) 分析算法

**LR(0)** 是最基础的自底向上语法分析算法。它不向前看任何符号，仅根据当前栈顶状态决定 shift 还是 reduce。

## 形式化定义

### LR(0) 项目的概念

详见 [[编译原理/关系图谱-节点内容/LR(0) 项目|LR(0) 项目]]。

对于产生式 `A → αβ`，LR(0) 项目为 `A → α·β`。圆点左边是已识别的部分，右边是待识别的部分。圆点在最后的项目 `A → α·` 称为**归约项目**（完整项目）。

### CLOSURE 运算

给定项目集 I，`CLOSURE(I)` 的定义：

```
CLOSURE(I)：
  repeat
    for each A → α·Bβ in I           —— 圆点后是非终结符 B
      for each B → γ in 文法
        把 B → ·γ 加入 I（如果尚不存在）
  until I 不再变化
```

**形式化解释**：如果当前状态"期望"接下来识别 B，那就必须考虑 B 推导出的所有可能性——把 B 的每条产生式的最初状态（圆点在最左）都拉进这个状态。

### GOTO 运算

```
GOTO(I, X) = CLOSURE( { A → αX·β | A → α·Xβ ∈ I } )
```

把状态 I 中所有圆点后紧跟 X 的项目圆点右移一位，再对结果求 CLOSURE。

### LR(0) 项目集规范族

从**增广文法**的初始项目 `S' → ·S` 出发，反复应用 GOTO：

```
I₀ = CLOSURE({ S' → ·S })
for each Iᵢ 和每个文法符号 X：
   if GOTO(Iᵢ, X) 非空且尚未在规范族中：
      将其加入规范族
```

得到的状态集合 `{I₀, I₁, ..., Iₙ}` 就是 LR(0) 项目集规范族（等价于识别活前缀的 DFA）。

## ACTION 和 GOTO 表的构造规则

对规范族中的每个状态 Iᵢ：

| 条件 | 表条目 |
|------|--------|
| 存在项目 `A → α·aβ ∈ Iᵢ`，且 `GOTO(Iᵢ, a) = Iⱼ`（a 是终结符） | `ACTION[i, a] = shift j` |
| 存在项目 `A → α· ∈ Iᵢ`（归约项目） | **对所有**终结符 a（包括 $）：`ACTION[i, a] = reduce A→α` |
| 存在项目 `S' → S· ∈ Iᵢ` | `ACTION[i, $] = accept` |
| `GOTO(Iᵢ, A) = Iⱼ`（A 是非终结符） | `GOTO[i, A] = j` |

> **关键点**：归约项目的 ACTION 条目对**所有**终端填入 reduce，完全不看向前符号。这是 LR(0) 产生 shift/reduce 和 reduce/reduce 冲突的根本原因。

## 冲突的形式化定义

### [[移进-归约冲突]]

同一个状态 I 中同时存在：
- 移进项目 `A → α·aβ`（需要 `ACTION[i, a] = shift j`）
- 归约项目 `B → γ·`（需要 `ACTION[i, a] = reduce B→γ`）

两个要求冲突。

### [[归约-归约冲突]]

同一个状态 I 中存在两个不同的归约项目 `A → α·` 和 `B → β·`。它们对同一输入符号都要求归约，但归约的产生式不同。LR(0) 无法区分选哪条。

## 为什么 LR(0) 只能处理极简单的文法

LR(0) 的归约策略是**盲目的**：只要状态中有完整项目，就对所有符号归约。它等价于问「这个状态是否允许我在不读书的情况下判断句子完了？」——大多数实际语言的文法答案是「不」。

## 例题

考虑文法：

```
E → E + T | T
T → id
```

增广文法加 `E' → E`。从 `I₀ = CLOSURE({E' → ·E})` 出发构造规范族（此处省略完整的状态列表）。在某个状态中会同时出现 `E → T·`（归约）和 `T → id·`（归约）——如果它们出现在同一状态，就产生 r/r 冲突。
