# 依赖图与属性计算顺序

> 属性方程说「怎么算」，依赖图说「先算谁、后算谁」。

---

## 一、为什么需要依赖图

属性文法给出了每条语义规则，但编译器需要知道**按什么顺序执行这些规则**。

例如：

```
number₁.val = number₂.val × 10 + digit.val
```

必须先知道 `number₂.val` 和 `digit.val`，才能计算 `number₁.val`。

**核心问题**：找一个合法的求值顺序，使每个属性在被使用前已经计算完成。

---

## 二、依赖图的定义

对于每条语义规则：

```
X.a = f(Y.b, Z.c)
```

说明 `X.a` 依赖于 `Y.b` 和 `Z.c`。

在依赖图中画边：

```
Y.b ──→ X.a
Z.c ──→ X.a
```

箭头方向：**被依赖的属性 → 依赖它的属性**（先算的 → 后算的）。

---

## 三、例子：数字求值

### 文法与规则

```
number₁ → number₂ digit
  number₁.val = number₂.val × 10 + digit.val
```

### 依赖关系

```
number₂.val ──→ number₁.val
digit.val   ──→ number₁.val
```

### 对于 345 的依赖链

```
digit(3).val
    ↓
number(3).val ────┐
digit(4).val      │
    ↓              ↓
number(34).val ───┐
digit(5).val      │
    ↓              ↓
number(345).val
```

### 合法计算顺序

```
digit(3).val
number(3).val
digit(4).val
number(34).val
digit(5).val
number(345).val
```

综合属性天然自底向上，顺序和依赖图的方向一致。

## 四、例子：声明类型传播

### 文法与规则

```
decl → type var-list
  var-list.dtype = type.dtype

var-list₁ → id , var-list₂
  id.dtype = var-list₁.dtype
  var-list₂.dtype = var-list₁.dtype

var-list → id
  id.dtype = var-list.dtype
```

### 依赖关系

```
type.dtype
    ↓
var-list.dtype
    ↓
id.dtype  ← var-list₂.dtype
```

类型信息从 `type` 出发，沿着 `var-list` 一路传递到每个 `id`。

---

## 五、循环依赖

如果属性方程形成环：

```
A.x = B.y + 1
B.y = A.x + 1
```

依赖图出现：

```
A.x ←→ B.y
```

有环的依赖图无法找到合法的顺序计算。这种属性文法需要更复杂的求解策略（如不动点迭代），而不只是简单的顺序求值。

---

## 六、依赖图的意义总结

| 概念 | 回答的问题 |
|------|-----------|
| 属性方程 | 属性**怎么算** |
| 依赖图 | 属性**按什么顺序算** |

| 图的性质 | 含义 |
|----------|------|
| 无环 | 可以拓扑排序，找到合法计算顺序 |
| 有环 | 属性相互依赖，需要迭代或无法直接计算 |
