# LR 系列语法分析对比总览

> 不是特点列表。是每种方法的**形式化定义、构造算法、表是怎么来的**。

---

## 零、前置概念：LR 分析器的结构

**LR(k)** = 自底向上、从左到右扫描（L）、最右推导的逆（R）、向前看 k 个符号。

一个 LR 分析器 = 两张表 + 一个栈：

| 组件 | 定义 |
|------|------|
| **ACTION 表** | `ACTION[state, terminal]` → shift / reduce / accept / error |
| **GOTO 表** | `GOTO[state, non-terminal]` → 下一个状态 |
| **分析栈** | 存状态号 + 文法符号对：`s₀X₁s₁X₂s₂...` |

一次分析循环：

```
设当前状态 = 栈顶状态 s，当前输入 token = a

查 ACTION[s, a]：
  - shift s'  → 压入 a 和状态 s'，读下一个 token
  - reduce A→β → 弹出 |β|×2 个元素，设新栈顶为 t
                压入 A 和 GOTO[t, A]
  - accept    → 分析成功
  - error     → 语法错误
```

## 一、LR(0)：定义与构造

### 1.1 LR(0) 项目的形式化定义

给定产生式 A → αβ（α、β 可以是符号串），一个 **LR(0) 项目**是：

```
A → α · β
```

圆点 `·` 表示"已经识别了 α，接下来期望识别 β"。一条产生式有 |αβ| + 1 个项目，圆点可以在任意两个符号之间、最前或最后。

**归约项目**（点在最后）：A → α ·——表示可以把 α 归约为 A。

### 1.2 构造 [[编译原理/关系图谱-节点内容/项目集规范族|LR(0) 项目集规范族]]

两个核心运算（详见 [[编译原理/关系图谱-节点内容/闭包运算|CLOSURE]] 和 [[编译原理/关系图谱-节点内容/跳转函数|GOTO]] 节点）：

**CLOSURE(I)**——若状态 I 中存在项目 A → α · Bβ（圆点后是非终结符 B），则把 B 的所有产生式以 `B → ·γ` 形式**加进 I**，重复直到不再增加：

```
CLOSURE(I)：
  repeat
    for each A → α·Bβ in I
      for each B → γ in 文法
        把 B → ·γ 加入 I
  until I 不再变化
```

**GOTO(I, X)**——把状态 I 中所有圆点后紧接符号 X 的项目，圆点右移一位，再对结果求闭包：

```
GOTO(I, X) = CLOSURE( { A → αX·β | A → α·Xβ ∈ I } )
```

从初始项目 S' → ·S 出发，反复对每个状态和每个符号应用 GOTO，直到不再产生新状态，得到**LR(0) 项目集规范族**（一个 DFA）。

### 1.3 从规范族构造 ACTION 和 GOTO 表

对每个状态 Iᵢ：

| 条件 | 动作 |
|------|------|
| 存在项目 A → α·aβ ∈ Iᵢ，且 GOTO(Iᵢ, a) = Iⱼ | `ACTION[i, a] = shift j` |
| 存在归约项目 A → α· ∈ Iᵢ | 对**所有**终结符 a：`ACTION[i, a] = reduce A→α` |
| 存在项目 S' → S· ∈ Iᵢ | `ACTION[i, $] = accept` |

对每个非终结符 A，若 GOTO(Iᵢ, A) = Iⱼ：`GOTO[i, A] = j`。

**LR(0) 的致命问题**：归约项目出现时，对**所有输入符号**都填 reduce，完全不看向前符号。一旦同一个状态里同时有移进和归约项目，表条目冲突（shift/reduce conflict）。这就是它"太莽"的形式化原因。

## 二、SLR(1)：定义与改进

### 2.1 FOLLOW 集的形式化定义

首先回顾：对非终结符 A：

```
FOLLOW(A) = { a | 存在推导 S ⇒* αAaβ, 其中 a 是终结符 }
           ∪ { $ | 存在推导 S ⇒* αA }
```

一句话：文法允许紧跟在 A **之后**出现的所有终结符，加上结束符 $（如果 A 可以出现在句子末尾）。

> FIRST 集定义见 [[编译原理/关系图谱-节点内容/Follow 集合]]。

### 2.2 SLR(1) 只改了一条规则

SLR(1) 沿用 LR(0) 完全相同的 DFA（项目集规范族）。唯一的改动在填 ACTION 表时：

| LR(0) | SLR(1) |
|-------|--------|
| 归约项目 A → α· 出现 → 对**所有** a 填 reduce | 归约项目 A → α· 出现 → 只对 a ∈ **FOLLOW(A)** 填 reduce |

形式化：

```
对状态 Iᵢ 中的每个归约项目 A → α·：
  for each a ∈ FOLLOW(A)：
    ACTION[i, a] = reduce A→α
```

如果 `{a | A → α· ∈ Iᵢ 且 a ∈ FOLLOW(A)}` 与 `{a | B → β·aγ ∈ Iᵢ}` 不交，则移进-归约冲突被消除。

### 2.3 SLR(1) 仍然不够精确

FOLLOW(A) 是**全局**的——只要文法中某处 A 后面可以跟 a，就算 a ∈ FOLLOW(A)。但在**特定状态 Iᵢ** 里，A 后面实际可能跟着的符号集合可能比 FOLLOW(A) 小。SLR(1) 用全局 FOLLOW 替掉了上下文，有时候填了不该填的 reduce，造成虚假冲突。

## 三、LR(1)：形式化定义

### 3.1 LR(1) 项目

**LR(1) 项目**的形式：

```
[A → α · β, a]
```

其中 `a` 是**前看符号（lookahead）**，含义是：

> "如果将来用 A → αβ 完成归约，归约到 A 的那一刻，下一个输入必须是 a。"

这个 a **不是**用来指导圆点后面的解析——圆点后面的部分 β 仍然按 CFG 文法匹配。a 只在**归约发生的那一刻**才出场：它允许 parser 在所在状态下、看到输入 a 时把 αβ 归约为 A。

### 3.2 LR(1) 的 CLOSURE

```
CLOSURE(I)：
  repeat
    for each [A → α·Bβ, a] in I：
      for each B → γ in 文法：
        for each b ∈ FIRST(βa)：     ← 关键：前看符号在这里传播
          把 [B → ·γ, b] 加入 I
  until I 不再变化
```

解释：当状态里有 `[A → α·Bβ, a]`，意味着接下来要识别 B。B 推导完成后，parser 面对的第一个符号来自 `FIRST(βa)`。如果 B 推导到空（B ⇒* ε），parser 直接面对 a。所以 B 的所有产生式的**前看符号** = `FIRST(βa)`，而不是照搬 a。

### 3.3 LR(1) 的 GOTO

```
GOTO(I, X) = CLOSURE( { [A → αX·β, a] | [A → α·Xβ, a] ∈ I } )
```

GOTO 不改变 lookahead——前看符号只通过 CLOSURE 传播。

### 3.4 LR(1) 的 ACTION 表构造

```
对状态 Iᵢ 中的每个 [A → α·aβ, b]：
  if GOTO(Iᵢ, a) = Iⱼ：ACTION[i, a] = shift j

对状态 Iᵢ 中的每个 [A → α·, a]：
  ACTION[i, a] = reduce A→α    ← 只对项目自己的 lookahead 归约！

对 [S' → S·, $]：
  ACTION[i, $] = accept
```

关键差异：每个归约项目有自己的 lookahead，不是全局 FOLLOW。同一个产生式 A → α 在不同上下文（不同状态）里，可能带不同的前看符号集合，从而只在**确实合理**的输入下触发归约。

### 3.5 LR(1) 为什么状态多

FIRST(βa) 因 β 不同而变化，导致同一产生式 B → γ 在不同上下文里以不同的 lookahead 进入项目集。LR(0) 里核心相同的项目在 LR(1) 里可能因为 lookahead 不同而分裂成多个状态。状态数可能膨胀到 LR(0) 的 10 倍以上。

## 四、LALR(1)：合并核心相同的 LR(1) 状态

### 4.1 定义

将 LR(1) 状态中**核心（去掉 lookahead 后的 LR(0) 部分）相同**的状态合并为一个，并把 lookahead 取并集。

```
两个 LR(1) 状态 I 和 J 满足：
  core(I) = core(J)    →    合并为 K = core(I) with lookaheads(I) ∪ lookaheads(J)
```

### 4.2 合并后的影响

- **状态数** = LR(0) 的状态数（大幅压缩）
- **移进能力**：完全保留——因为移进不依赖 lookahead
- **归约能力**：lookahead 是多个 LR(1) 状态的并集，比单一 LR(1) 状态宽松
- **可能引入新的 reduce/reduce 冲突**：两个原本在不同 LR(1) 状态下分别安全的归约，合并后 lookahead 取并，可能重叠

### 4.3 能力层级

```
LR(0)  ⊂  SLR(1)  ⊂  LALR(1)  ⊂  LR(1)
  ↑         ↑          ↑          ↑
 最小      加FOLLOW   合并LR(1)   每个项目自带lookahead
```

**Yacc / Bison 使用 LALR(1)**：同时拿到了接近 LR(1) 的识别能力和接近 SLR(1) 的表规模。

## 五、四种方法的精确定义对比

| 方法 | 项目形式 | 前看符号来源 | CLOSURE 算法 | ACTION 归约条件 | 状态数 |
|------|----------|------------|------------|--|:--:|
| **LR(0)** | `A → α·β` | 不使用 | 基于是否有 · 后 NT | 对所有终结符都归约 | 最少 |
| **SLR(1)** | `A → α·β` | FOLLOW(A) | 同 LR(0) | 输入 ∈ FOLLOW(A) | 同 LR(0) |
| **LR(1)** | `[A → α·β, a]` | FIRST(βa) 传播 | lookahead 参与闭包 | 输入 == 项目 lookahead | 最多 |
| **LALR(1)** | `[A → α·β, a]` | 合并 LR(1) 的 lookahead | 同 LR(1) | 输入 ∈ 合并后的 lookahead 集 | 同 LR(0) |

---

## 六、最简记忆

> **LR(0)**：项目是 `A → α·β`，归约不看终端，冲突多。
> **SLR(1)**：仍用 LR(0) 状态，但归约只对 FOLLOW(A) 中的终端进行。
> **LR(1)**：项目是 `[A → α·β, a]`，a 从 `FIRST(βa)` 传播进闭包。归约只对 a 进行。
> **LALR(1)**：合并 LR(1) 核心相同的状态，能力够，表不大。Yacc 用这个。
