# 图谱剧情巡检 (Graph Plot Inspection)

> **"故事不是一条线，是一张网。巡检就是在网中寻找断裂的节点。"**

**图谱剧情巡检**利用 Obsidian 双向链接天然形成的知识图谱，对整部小说的结构进行**自动化逻辑体检**。

---

## 为什么需要巡检？

传统写作中，作者靠"感觉"来判断情节是否连贯。但当小说超过 30 万字后，人类的短期记忆无法覆盖所有设定——这就是**长篇小说熵增**的核心问题。

图谱巡检将"直觉"转化为**可计算的图论问题**：

| 直觉问题 | 图论等价问题 |
|---------|------------|
| "这个角色是不是被遗忘了？" | 节点出度/入度变化率检测 |
| "这条伏笔有没有收？" | 从设置点到回收点的路径存在性 |
| "是不是有两个地方逻辑矛盾？" | 属性值在全图中的一致性 |
| "有没有孤立的情节碎片？" | 连通分量检测 |

---

## 子系统

| 子系统 | 检测内容 | 算法思路 |
|--------|---------|---------|
| [[节点连通性检测]] | 角色/事件/地点的活跃度与出度入度 | 度分布统计 + 滑动窗口异常检测 |
| [[因果链追溯]] | 从结局反向 BFS，验证每条因果路径 | 有向图反向搜索 + 路径完整性校验 |
| [[事件孤岛预警]] | 检测未接入主线的独立事件簇 | 连通分量分析 + 弱连通分量标记 |

---

## 巡检报告格式

```yaml
inspection:
  timestamp: "2026-04-30T15:30:00Z"
  scope: "第1章-第30章"
  findings:
    - type: "character_dormant"
      node: "[[角色·陈雪]]"
      severity: "warning"
      detail: "在过去 15 章中出度为 0，最后一次出现于第 14 章"
      suggestion: "考虑在第 31 章安排陈雪回归或增加侧面提及"
    - type: "timeline_conflict"
      node: "[[事件·帝都沦陷]]"
      severity: "error"
      detail: "事件标注时间为 永安三年三月，但引用它的第 28 章设定在 永安二年十二月"
    - type: "orphan_event"
      node: "[[事件·古庙密谈]]"
      severity: "info"
      detail: "该事件节点仅被 1 个章节引用，且与主线无因果连接"
```

## 技术架构

```
Obsidian Vault (.md 文件)
        │
        ▼
Wiki-link 解析器 → NetworkX 有向图构建
        │
        ▼
   ┌────────────┐
   │ 巡检管道     │
   ├────────────┤
   │ 1. 度分析   │ → 角色活跃度报告
   │ 2. 路径分析 │ → 因果链完整性
   │ 3. 分量分析 │ → 事件孤岛检测
   │ 4. 时序分析 │ → 时间线一致性
   └────────────┘
        │
        ▼
   巡检报告 JSON → Next.js 前端热力图渲染
```
