# 三维空间交互平滑：弹簧阻尼轨迹推算与悬停防抖缓冲

我们在使用 30 FPS 的普通 USB 摄像头进行手势交互时，经常会遇到两大痛点：一是摄像头帧率低、噪点大，导致 3D 关节点和指针在 60Hz+ 的显示器上看起来有明显的**阶跃感（Juddering）与微抖动**；二是人在悬停（Hover Dwell）确认时，手尖天然的微抖会让光标频繁滑出大陆边缘，导致**确认进度条不断被瞬间清零**。

为了解决这些体验细节，我们为以太空间交互系统引入了**物理模拟外推**、**状态迟滞缓冲区**以及**摄像头生命周期绑定**。

---

## 1. 弹簧阻尼物理平滑（Spring-Damper Extrapolation）

为了消除坐标突变产生的 twitchy（神经质）观感，我们引入了经典的**弹簧-阻尼模型（Hooke's Law with Damping）**。我们不再把指针强行拉向目标，而是假设指针与目标之间连着一根隐形的弹簧，以屏幕真实的渲染帧率（$\Delta t$）迭代计算：

```javascript
const stiffness = 170; // 劲度系数（较低值能更好过滤手部高频震颤）
const damping = 26;    // 阻尼系数（较高值让光标平稳收拢，防止晃荡）

const forceX = -stiffness * (current.x - target.x) - damping * velocity.x;
const forceY = -stiffness * (current.y - target.y) - damping * velocity.y;

velocity.x += forceX * dt;
velocity.y += forceY * dt;

current.x += velocity.x * dt;
current.y += velocity.y * dt;
```

### 调优参数
* **2D 遥感光标**（`stiffness: 170`, `damping: 26`）：光标不再跟随手尖神经质地抖动，而是呈现出一种如 Vision Pro 般的深邃阻尼感，平稳滑行。
* **3D 全息骨骼关节**（`stiffness: 150`, `damping: 24`）：21 个三维关节点被赋予了更柔和的引力，过滤掉摄像头由于低采光产生的关节点跳变，骨架摆动流畅如液态金属。

## 2. 悬停防抖缓冲与防误触（Hover Dwell Grace Period）

手尖稍微一抖，滑落到“海洋”哪怕 1 帧，悬停计时器就会重置。对此，我们通过 `useRef` 状态与组件渲染解耦，实现了**防抖悬停缓冲区**，并调整参数以防止误触：

```javascript
const GRACE_PERIOD = 250; // 250ms 的抖动宽容期

// 1. 只要手尖在有效大陆上（排除 ocean 干扰），正常累加悬停确认进度
if (currentActive && currentActive !== 'ocean') {
  state.graceTimeRemaining = 0;
  // 增加确认时间到 1.6 秒 (dt / 1600)，要求更明确的悬停意图，防止扫过时误触
  state.progress = Math.min(100, state.progress + (dt / 1600) * 100);
} else {
  // 2. 抖动偏离时启动 250ms 宽容倒计时
  if (state.currentTargetBiome) {
    if (state.graceTimeRemaining === 0) {
      state.graceTimeRemaining = GRACE_PERIOD;
    }
    state.graceTimeRemaining -= dt;
    
    if (state.graceTimeRemaining <= 0) {
      // 3. 超过 250ms 未移回，以 250ms 的速度快速平滑漏干进度条，干净利落
      state.currentTargetBiome = null;
      state.progress = Math.max(0, state.progress - (dt / 250) * 100);
    }
  } else {
    state.progress = Math.max(0, state.progress - (dt / 250) * 100);
  }
}
```

## 3. 解决跨页面生命周期：回调 Ref 重启机制

### 痛点描述
手势控制的 Provider（`HandTrackingProvider`）包裹在 React 最外层（属于单例全局组件）。当用户从首页（`/`）跳转到博客（`/blog`），再切回首页时：
1. 首页的 `<video>` 和 `<canvas>` 节点被**彻底销毁并重新挂载**。
2. 内部 Ref `.current` 重新指向新节点，但由于 ref 变化不触发 re-render，Provider 内部的 MediaPipe 线程仍在读取已经销毁的旧节点，造成**摄像头黑屏、骨骼画面卡死**。

### 回调 Ref（Callback Ref）解决方案
在 `useHandTracking.js` 中弃用普通的 `useRef` 对象，改用**函数式回调 Ref** 配合计数器触发器（`elementTrigger`）：

```javascript
// 1. 在 Provider 中定义状态触发器与回调 Ref
const [elementTrigger, setElementTrigger] = useState(0);

const videoRefCallback = useCallback((node) => {
  videoRefInternal.current = node;
  setElementTrigger((prev) => prev + 1); // 节点挂载/卸载时，强制累加计数器
}, []);
```

在主副作用 `useEffect` 中，将 `elementTrigger` 放入依赖数组。一旦节点经历切换：
1. **自动清理（Cleanup）**：销毁上个页面的 MediaPipe 相机环、停止底层 Webcam 视频流。
2. **零功耗休眠**：在非 3D 页面（如 `/blog`），由于没有 `<video>` 节点挂载，`videoRefInternal.current` 为 `null`，相机系统**直接退出初始化，停止消耗任何 CPU 资源**。
3. **即时重装（Re-bind）**：重回 3D 页面时，新节点挂载触发 `elementTrigger` 改变，相机自动申请硬件流并重新绑定新组件，实现免刷新无缝重连。
